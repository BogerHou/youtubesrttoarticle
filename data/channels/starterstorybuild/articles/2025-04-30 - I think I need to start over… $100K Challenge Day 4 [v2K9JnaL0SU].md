# 我差点把项目推倒重来，但先做完了支付

这一天，我继续推进自己的 10 万美元挑战：只用 AI 工具做产品，并公开记录整个过程，直到它赚到 10 万美元。

项目还是 AI Thumbnails Maker，一个用 AI 生成 YouTube 缩略图的工具。前几天我已经把基础页面和一些交互做了出来，但越往后推进，越发现真正难的不是“把界面做出来”，而是把一个能收费、能生成、能发布的产品串起来。

这一天的重点很明确：接入 Stripe 支付，调整功能优先级，砍掉暂时不该做的功能，并给自己设置一个必须发布的截止日期。

结果后半段，我差点决定全部推倒重来。

## 先把支付跑通，哪怕只是测试环境

我先列了一个很简单的待办清单。没有复杂项目管理工具，就写在 Apple Notes 里。

排在最前面的，是支付。

原因很简单：这个产品调用 AI 图片生成 API（应用程序接口）是有成本的。如果用户每次生成缩略图都要消耗我的额度，那就不能一直免费开放。最基础的商业逻辑应该是：用户先付 5 美元，然后才能生成缩略图。

我新建了 Stripe 账号，先使用沙盒环境。（**沙盒环境是支付系统的测试模式，可以模拟付款成功或失败，不会真的扣钱。**）实现方式选择 Stripe Checkout。（**Stripe Checkout 是 Stripe 提供的托管支付页，开发者不用自己处理完整银行卡表单和支付安全细节。**）

接下来，我把需求写给 Cursor，让它把 Stripe Checkout 接进项目。

第一次当然报错。我复制错误信息，继续丢回 Cursor。大概两轮提示后，支付流程跑通了。

我用 Stripe 的测试卡号 `4242 4242 4242 4242` 完成了一次模拟付款。页面跳转到 `localhost:3000/stripe/success`，Stripe 后台也显示了一笔假的 5 美元付款。

这一步很小，但很关键。

很多人做产品会先沉迷功能，最后才想收费。但我越来越觉得，支付应该尽早接进去。哪怕只是测试模式，它也会逼你思考：用户到底为什么付钱？什么时候付？付完之后得到什么？

## 不要让用户去 ChatGPT 里“手动完成产品”

前一个版本里，我的流程还有点别扭：用户在工具里做完一些设置后，可能还要去 ChatGPT 里继续操作。

这不是一个真正完整的产品。

如果用户要在多个工具之间来回复制提示词、上传图片、等结果，那我只是做了一个“提示词包装器”，而不是一个好用的缩略图生成器。

所以我决定调整方向：图片必须在后端生成。用户付费后，产品自己调用 OpenAI 的图片 API，直接生成结果。

当时我在等 GPT-4o 图片能力相关的 API 权限，希望能把模型热切换进去。（**热切换指的是在不重写整套产品的情况下，把底层模型或服务替换成新的版本。**）

这也意味着，我不能无限加功能。我要先把最短路径跑通：用户进入页面，输入需求，付款，然后生成缩略图。

## 首页不再像普通 SaaS，而是直接进入画布

我还改了落地页。

参考对象是朋友 Danny Postma 的 AI 证件照服务。他的页面很直接：展示前后对比、展示可信度、告诉用户能得到什么结果。

所以我也加入了社会证明。（**社会证明指的是用用户案例、数据、评价、Logo、成果截图等方式降低新用户的不信任感。**）比如 before / after 对比图、Logo 占位、使用案例等。早期没有用户案例，我就先用自己 YouTube 频道的缩略图做示例。

但我不想让首页只是一个传统 SaaS（软件即服务）介绍页。

我把首页改成更像一个空白画布。用户一打开，就能开始创作缩略图，而不是先读一大段营销文案。

这类 AI 工具的关键不是讲太多，而是让用户尽快体验“我真的能做出东西”。

## 没准备好之前，先加访问保护

在产品还没准备好之前，我加了基础认证。

原因不是神秘，而是成本。

如果网站提前被人访问，用户乱点生成按钮，就可能直接消耗 OpenAI 额度。一个还没准备好的产品，不应该无保护地挂在公网。

随后我开始连接域名 `aithumbnailsmaker.com`。流程涉及 Namecheap、Cloudflare、Heroku 和 GitHub。

域名解析这类工作很琐碎。我直接问 ChatGPT：nameserver 应该怎么设置，Cloudflare 和 Heroku 怎么连接。然后一步步照做。

到这里，产品已经越来越接近“能上线”的状态。

## 我用一个赌约逼自己发布

我还给自己设了一个很硬的发布截止日期。

在一次朋友聚会中，我和朋友打赌：如果月底还没发布，就给他 1000 美元。后来另一个朋友也加入，风险变成了 2000 美元。

这不是一个推荐所有人模仿的方法，但对我有用。

因为没有外部压力时，产品很容易一直停在“再改一点”的状态。尤其是 AI 编程工具太好用了，你总觉得还可以再加一个功能、再优化一个页面、再调整一个流程。

但真正重要的是发布。

## 接入图片 API 后，问题开始变复杂

后来，我终于拿到了 OpenAI 图片 API 的访问权限。

我把 OpenAI key 放进环境变量，安装 Ruby 相关的 OpenAI gem，增加 thumbnails controller 和 route，然后开始让后端生成图片。

一开始还是各种 server error。我继续把错误复制给 Cursor，让它修。

很快我发现一个问题：Cursor 有时会参考旧文档，甚至调用旧模型。于是我必须明确告诉它使用新模型 `gpt-image-1`，不要再按旧的 DALL-E 写法来处理。

最终，图片生成成功了。

但新问题也出现了：成本。

单张图片大约要 0.36 美元。如果一次生成 4 张，成本可能在 1.25 到 1.50 美元之间。那我就必须认真设计定价。5 美元看起来不贵，但如果用户多次生成、失败重试、图片尺寸更高，利润可能很快被吃掉。

AI 产品看起来像软件，实际上经常带着“每次使用都要花钱”的成本结构。这和传统 SaaS 不一样。

## 把生成动作放到付费墙后面

下一步，我把 OpenAI 调用放到付费墙后面。（**付费墙就是用户必须先付费，才能访问某个功能或内容。**）

流程应该是：用户填写缩略图需求，进入 Stripe 支付，支付成功后再生成图片。

第一次实现时，Cursor 把图片数据塞进 session，结果触发 cookie overflow。（**cookie overflow 指浏览器 cookie 或 session 存储的数据太大，超过限制，导致请求失败。**）这说明图片数据不能这么存。

于是我改成更合理的方式：用数据库和 Active Storage 一类的机制保存生成结果，并建立 Thumbnail model 来管理缩略图记录。

这一步终于让核心流程跑通了：先付款，再生成。

## Vibe coding 的边界也暴露出来了

真正让我崩溃的是后面。

我想继续加四张不同风格预设、后台任务、更复杂的生成流程。（**后台任务指不在用户请求当下同步完成的任务，比如图片生成、邮件发送、视频处理，通常放到队列里慢慢执行。**）

结果 Cursor 开始越改越乱。

它不断 hallucinate。（**在 AI 编程里，hallucinate 通常指模型编造不存在的 API、方法、文件或错误结论。**）有时引用旧的 DALL-E 模型，有时补一个补丁又引出另一个 bug，最后代码像一层层胶带贴出来的。

这就是 vibe coding 的现实边界。

用 AI 起项目、做页面、搭 CRUD、接基础 API，速度非常快。但当你开始处理较新的 API、复杂状态、后台任务、支付后流程、文件存储和错误恢复时，AI 很容易开始绕圈。

如果你不理解系统结构，只是不断把报错贴回去，项目会越来越难收拾。

## 我没有放弃，但决定先停一天

到最后，我真的有一刻想把项目全部删掉，从头开始。

但我没有这么做。

我决定先停 24 小时，第二天再回来。因为当代码和情绪一起变乱时，继续硬推往往只会制造更多问题。

这一天给我的最大经验是：AI 能让你更快进入产品阶段，但它不会替你做产品判断。

你仍然要决定先做什么，砍掉什么，什么时候收费，成本怎么算，什么时候停止加功能，什么时候发布。

我还在挑战里，项目也还没有完成。但至少这一天我确认了一件事：支付、域名、API、成本和发布压力，才是真正把一个 AI demo 变成产品的地方。
