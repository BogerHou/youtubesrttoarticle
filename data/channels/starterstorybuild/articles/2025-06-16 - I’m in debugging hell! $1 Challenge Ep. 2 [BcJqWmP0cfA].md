# 我在调试地狱里，把走路 App 推进了一步

这一期的一美元挑战，我几乎整集都卡在一个地方：调试。

我是一个完整的新手，目标也很简单：用 ChatGPT、Cursor 这类 AI 编程工具，做出一个移动 App，并最终在网上赚到第一美元。

听起来很轻巧。

但真正打开项目之后，我很快就掉进了“调试地狱”。（**调试就是找出程序为什么报错、为什么不能按预期运行，并一步步修正它。对新手来说，最难的地方不是复制代码，而是不知道错误到底从哪里来。**）

屏幕上全是错误提示。我不知道它们具体是什么意思，只能截图、复制，然后丢给 ChatGPT。

ChatGPT 会建议我重启项目，把某段提示词交给 Cursor，或者重新安装某些东西。Cursor 也会继续尝试修复。

但结果经常是：修完一个，又冒出另一个。

最后我甚至开始怀疑：这个项目是不是应该推倒重来？

## 有时候，重来比硬修更快

这次最明显的教训是：AI 能帮你修很多问题，但它不一定能把一个已经乱掉的项目救回来。

ChatGPT 后来也给了一个判断：问题可能出在原生代码或依赖上，而且是那种“静默崩溃”的问题。（**依赖可以理解为项目要借用的外部工具包；原生代码则是更贴近手机系统底层的代码。移动 App 一旦牵涉到这些部分，错误往往比网页项目更难排查。**）

它的建议很直接：最干净的办法，可能是从一个新项目开始。

这听起来有点挫败，但对新手来说反而是一个重要判断。

不是所有错误都值得继续硬修。尤其当你已经不知道前面改过什么、安装过什么、删除过什么时，重新搭一个干净项目，可能比在旧项目里继续绕圈更有效。

于是，我开始了 `walking buddy 3`。

没错，已经是第三次了。

## 先写清楚需求，再让 AI 动手

这次我没有直接让 Cursor 开写。

我先用了一个叫 Code Guide 的工具。它不是赞助，我自己付了大约 200 美元年费。

它的作用不是替我写完整代码，而是帮我把项目想清楚：App 要有哪些组件、用户流程是什么、哪些地方需要同步数据、哪些页面先做、哪些可以以后再说。

它会根据问卷生成产品需求文档和流程图。（**产品需求文档就是把“我要做什么”写成工程师和 AI 都能理解的说明书；流程图则把用户点击路径画出来，减少边做边猜。**）

这一步对新手很关键。

如果我只对 Cursor 说“帮我做一个走路 App”，AI 很容易一次性生成太多东西。页面、权限、数据同步、导航、样式全部混在一起，一旦出错，我根本不知道是哪一层坏了。

这次我把 Code Guide 生成的文档和流程图拖进 Cursor，然后明确告诉它：

我是初学者。请按照这些项目需求来做，并且边做边解释你在做什么。

## 让项目分阶段长出来

Cursor 开始按阶段执行。

第一阶段是搭建基础 Expo 项目和开发环境。（**Expo 是 React Native 生态里常用的移动 App 开发框架，可以让开发者更快在手机上预览和调试。**）

第二阶段是做前端界面。

它安装依赖，创建欢迎页，补导航类型，建立主标签导航、根导航，再改 `app.tsx`。

第一次跑起来时，我终于看到了一个像样的页面：

`Welcome to Walking Buddy`

下面还有一句类似这样的文案：

`Track your steps, build healthy habits, and achieve your fitness goals.`

按钮写着 `Get started`。

虽然这个按钮一开始还没有真正逻辑，但至少项目不再只是报错窗口了。它有了入口。

这对新手来说很重要。

你需要先看到一点东西跑起来，才有继续做下去的信心，也更容易判断下一步该修哪里。

## 移动 App 最难的地方：健康数据

评论区有人提醒我，Expo Go 这个 App 可以一直保持运行，不需要每次都重新启动。

这确实帮我省了一些时间。

但真正的麻烦还在后面：同步健康数据。

我想让 App 读取 iPhone 的步数。结果很快遇到一个错误：`expo-health` 无法解析。

继续问 ChatGPT 后，我才知道问题可能不是我写错了，而是这个库已经不维护，或者和当前项目不兼容。

这里我学到一点：新手遇到依赖报错时，不要默认都是自己笨。

很多时候，是教程过时了、库不维护了、版本不兼容了。

最后的处理方式是：先卸载 `expo-health` 和 `expo-permissions`，暂时用占位逻辑继续往前走。（**占位逻辑就是先用假的或简化的数据占住位置，让界面和流程先跑通，真实能力以后再接上。**）

修完之后，欢迎页终于能进入健康数据授权页。

点击 `Get started`，会跳到 `Access health data`。

`Allow full access` 当时还没反应，但 `Continue without health data` 已经能继续往下走。

这不是完整成功，但它是一次推进。

## 先做仪表盘，再做习惯追踪

接下来我做了仪表盘。（**仪表盘就是把最关键的数据集中展示出来的页面，比如今天走了多少步、本周走了多少步、这个月走了多少步。**）

最初目标是展示三个数字：

今天步数、每周步数、每月步数。

我还想做一个类似 HabitKit 的网格。它看起来有点像 GitHub 的贡献图：每天一个小方块，绿色代表那天达到了 10,000 步。

一开始 Cursor 做出来的是日、周、月三个切换页。

但我真正想要的是三个数字同时出现在同一页。

于是我没有直接乱改代码，而是让 ChatGPT 帮我写一段更清楚的提示词，再交给 Cursor：

把日、周、月三个指标纵向堆叠显示在同一个页面里。

改完之后，页面算是能用了。它不完美，布局也没有完全居中，但方向对了。

随后我继续做习惯追踪页面。

最初是一个 30 天、7 列 5 行的网格。后来又加上说明：

`Each square equals 1 day. Green means you hit your goal.`

也就是每个方块代表一天，绿色表示当天完成目标。

再加上星期几和当前月份，页面终于开始像一个真实 App，而不是临时拼出来的 Demo。

## 最后一步：把进度保存下来

这集最后，我做了一件很工程化、但对新手非常重要的事：用 Git 和 GitHub 保存项目。

Git 是版本控制工具。（**你可以把它理解成项目的时间机器：每完成一段可用进度，就拍一次快照。以后出问题，可以知道自己改了什么，也能回到之前的状态。**）

我初始化了 Git 仓库，执行 `git add .`，然后做了第一次提交。

确认左侧没有未保存改动之后，我才算真正放心。

因为 AI 编程最大的问题之一，就是它改得太快。

如果没有版本记录，你很容易在一次看似简单的修复里，把前面好不容易跑通的东西弄坏。

## 这一天真正学到的东西

我在 X 上发了公开构建更新：

`Day 0.03`

大意是：今天还在调试炼狱里，但我终于走出了错误循环，加了几个小功能，今天比昨天好 1%。

这其实就是一美元挑战现在最真实的状态。

AI 编程很好玩，也确实降低了门槛。一个不会写代码的人，可以通过 ChatGPT、Cursor、Claude 这些工具，把一个移动 App 一点点推起来。

但移动 App 仍然复杂。

你会遇到依赖、权限、手机系统、数据同步、导航、真机测试。这些问题不是一句“让 AI 帮我写代码”就能全部消失。

这集给我的经验是：

不要一上来就让 AI 生成整个 App。

先写需求，画流程，拆阶段。

每次只推进一个小功能。

让 AI 解释它在做什么。

遇到错误时，不只是把报错复制进去，还要问：这个错误属于哪一层？是代码、依赖、环境，还是项目结构？

如果项目已经乱到看不清，重来并不可耻。

对新手来说，能从调试地狱里走出一步，就已经是在构建真正的能力。
