# 我在 AI 缩略图项目上踩了一个大坑：开工前没确认核心能力是否可用

这是 10 万美元 AI 构建挑战的第三天。

前两天，我做了一个叫 AI Thumbnails Maker 的项目：目标是让用户上传参考图、画草图、写提示词，然后生成 YouTube 缩略图。

听起来很合理。问题也真实存在：YouTube 缩略图很重要，做起来贵、慢、反复沟通。我自己也需要这个工具。

但第三天，我发现自己犯了一个很大的错误。

我开始构建之前，没有确认最核心的技术是否真的可用。

## 问题出在 GPT-4o 图像 API

我最初的灵感来自一个推文案例：用户画一张草图，上传给 GPT-4o，模型就能生成非常好的图像结果。

我想把这个能力接到自己的应用里，让用户在网页上完成整个流程。

但真正查文档和测试后，我发现 GPT-4o 的图像生成能力当时还不能通过 API 稳定访问。文档里写着，开发者很快可以通过 API（应用程序接口）生成图像，权限会在接下来几周逐步开放。

这就是时间差。

ChatGPT 里已经能用的能力，不代表开发者马上能通过 API 集成到自己的应用里。

我又测试了其他图像模型，但结果不够好。即使硬拼起来，也很难达到我想要的缩略图质量。

这让我意识到：产品最核心的承诺暂时无法自动化完成。

## 这就是平台风险

这件事也提醒我一个老问题：平台风险（**平台风险指你的产品依赖某个大平台或 API，一旦对方改规则、限制访问、涨价或关闭接口，你的业务就会被影响。**）。

OpenAI 已经有这个能力，但它选择什么时候开放 API，不由我决定。我的产品依赖它，所以我必须适应它的节奏。

这不是 OpenAI 独有的问题。所有依赖大平台的产品都会遇到：社交平台改算法，支付平台改政策，应用商店审核变严格，模型 API 价格上涨。

做创业时，平台能力可以让你快速起步，但也可能让你突然卡住。

## 我有两个选择

当时我大概有两个选择。

第一个选择，是继续硬做。用各种模型拼接，用不够理想的方案凑出一个自动生成缩略图功能。

这样做的问题是，输出质量很可能不够好。对缩略图工具来说，如果结果不好，用户不会付钱。糟糕的自动化不如不做。

第二个选择，是把能做的部分先做出来，把最后一步交给 ChatGPT 完成。

也就是说，我的应用负责让用户上传参考图、在图上画草图、添加文字标注、导出编辑后的参考图和高质量提示词。然后用户把这些内容复制到 ChatGPT 里生成缩略图。

这个方案很笨，也不够优雅。

但它有一个好处：用户最终可以拿到更好的缩略图结果。等 GPT-4o 图像 API 开放后，我再把最后一步自动接回来。

所以我选择了第二条路。

## 遇到限制，不等于项目立刻死亡

这一天最大的心理挑战，不是代码，而是接受项目变丑。

我原本想做一个完整工具：上传、画图、生成，一气呵成。现在变成了一个半自动工作流，需要用户复制图片和提示词，再跳到 ChatGPT。

这听起来不够酷，甚至有点尴尬。

但创业经常就是这样。你会遇到障碍，然后想出一个不完美但能继续前进的方案。

很多人一遇到这种情况就放弃。因为它破坏了原来的理想版本。

但真正要验证的不是“技术流程是否完美”，而是“用户是否愿意为了更好的缩略图使用这个流程”。如果用户愿意，那说明方向有价值；如果没人愿意，那我也能尽早知道。

## 我开始改产品功能

既然不能直接自动生成缩略图，我就把重点放到“生成高质量参考图和提示词”上。

我需要用户能做三件事：

第一，上传一张参考照片。

第二，在照片上画草图，比如指出背景、人物位置、要添加的元素。

第三，在照片上添加文字标注。

最后，把这张带标注的图片导出，再配合提示词交给 ChatGPT。

我打开 Cursor，找到生成缩略图区域的代码，然后用自然语言告诉它：我喜欢现在的绘图功能，但需要进一步支持用户上传参考图；上传后图片要显示在画布上，用户可以直接在图片上画，最后导出带标注的图片。

第一次没成功。图片上传后页面还是空白。

我继续提示：上传后应该把图片放到页面上，让我能在上面画。几轮之后，它终于能显示参考图，并允许我画线和导出。

## 加文字功能时，项目开始失控

下一步是添加文字。

我希望用户点击一个 “T” 按钮后，可以在图上选择位置，输入文字，并且能移动、调整。

我要求 Cursor 尽量使用成熟开源库，而不是手写一堆复杂逻辑。它先尝试使用 Fabric.js（**一个用于 HTML5 画布的 JavaScript 库，可以处理图片、绘图、文字对象等交互。**）。

但集成后，原本图片上传功能又坏了。

我开始调试。打开浏览器控制台（**浏览器控制台可以显示网页运行时的错误和日志，是调试前端问题的重要工具。**），查看上传图片时到底发生了什么。Cursor 也帮我加了日志。

但问题仍然没解决。

这时我进入了我称为“紧急模式”的状态。

## 紧急模式：让 AI 从头重新设计方案

当 AI 在一个坏方案上反复修补时，最好的办法不是继续让它补丁叠补丁，而是让它停下来，重新思考。

我切换到询问模式，对 Cursor 说：现在显然不行。如果你从零开始，会怎么实现？需求是上传图片、把图片放到页面上、支持绘图和添加文字，最好用最佳实践库，不要硬凑。

这次它建议改用 Konva.js（**一个更适合处理画布交互的 JavaScript 库，常用于图片编辑、拖拽、绘图、文本对象等场景。**）。

我让它重新实现。

结果这一次顺利多了。上传图片可以显示，绘图可以工作，文字可以添加，导出也能完成。

这给了我一个重要教训：用 AI 编程时，如果连续修不好，不要一直在坏架构上打补丁。让它重新提出方案，甚至换库、换结构，常常更快。

## 最后一步：生成提示词和手动工作流

功能能导出后，我又加了一个弹窗。

用户点击“生成缩略图”后，弹窗会告诉他们：目前我们正在等待 OpenAI 开放更合适的图像 API；为了得到更好的缩略图，请复制编辑后的图片和提示词，到 ChatGPT 中生成。

我让 Cursor 把提示词放进一个单独的 Markdown 文件，方便以后反复优化。因为这个产品短期内最核心的价值，可能就是“更懂 YouTube 缩略图的提示词”。

我还加了一个“额外细节”输入框，让用户写下更多要求，比如背景、人物动作、风格、文字等。这些内容会被合并进最终提示词。

最后，我把改动提交并部署到线上。

这不是我最开始想象中的完美产品，但它已经能完成一个实际流程：

上传参考图。

在图上画草图和标注。

添加额外说明。

导出图片。

复制提示词。

交给 ChatGPT 生成缩略图。

## 这一天最大的教训

第一，开工前必须确认核心技术是否可用。尤其是依赖 API 的产品，不能只看演示效果，要确认开发者能不能调用。

第二，ChatGPT 里能做，不代表你的应用里能做。产品能力和 API 能力是两回事。

第三，不完美方案也能推进验证。只要用户最终能得到价值，流程丑一点也可以先试。

第四，AI 编程会出很多 bug。学会看浏览器控制台、读日志、缩小问题范围，非常重要。

第五，当 AI 越修越乱时，让它从零重新设计，不要无休止补丁。

第六，创业里没有容易项目。你以为最简单的地方，往往就是第一堵墙。

这次我确实搞砸了：没有提前验证核心 API。

但我没有停。产品变得更笨、更绕，但仍然向前走了一步。接下来要做的，是把域名、支付、提示词质量、安全检查和发布流程继续补上。

真正的挑战，不是永远不犯错，而是犯错之后还愿意继续把它做出来。
