# 一美元挑战第一天：我用 AI 做了一个最小走路 App

这是 Gus 的一美元挑战第一集。

目标很简单：作为一个几乎没有开发经验的人，用 AI 想点子、做 App、营销、找用户，最后赚到网上的第一美元。

这一天，他没有做出一个完整产品。

但他做出了一个能跑起来的 MVP。

MVP 是最小可行产品。（**它不是最终产品，而是用最少功能验证核心想法的版本。**）

对一个真正的新手来说，这一步比想象中重要。因为真正的难点不是把功能想得多漂亮，而是把一个项目从“我有个想法”，推到“它真的能在手机上打开”。

## 一开始，先把想法砍小

Gus 最初想做一个走路 App。

最早的想法是：记录你长期走路的进度，比如“我正在用每天步数横穿美国”。App 会显示你离这个大目标还有多远。

这个想法有趣，但对新手来说有点大。

后来他看到一个爆火推文：有人想要一个极简财务 App，只显示三个数字：今天花了多少钱、本周花了多少钱、本月花了多少钱。

Gus 把这个想法迁移到走路场景：

我想要一个极简走路 App，只显示三个数字：今天走了多少步、本周走了多少步、本月走了多少步。

这就是很好的缩小范围。

很多新手的第一个产品失败，不是因为点子不好，而是第一版太大。想做地图、目标、成就、社交、分享、排行榜、健康数据同步，最后什么都没做完。

这一版只做三个数字。

这就够了。

## 用 ChatGPT 先写 PRD，而不是直接开写

Gus 一开始并没有马上打开 Cursor 写代码。

他先打开 ChatGPT，告诉它自己要做什么：

这是一个 step-counting app，也就是计步 App。

核心功能是三个大数字：今天、本周、本月。

界面要干净、极简，所有东西最好一屏就能看完。

他还把一些参考截图发给 ChatGPT，尤其是那些极简财务 App 的界面。

这里有个细节很实用：他用 Whisper Flow 语音输入，而不是手打。（**Whisper Flow 是语音转文字工具，适合把想法快速说出来，再交给 AI 整理。**）

因为自己是新手，他还特别提醒 ChatGPT：我只大概知道 ChatGPT、Cursor 和 Expo Go。

Expo Go 是一个用于预览 React Native App 的工具。（**React Native 是用 JavaScript / React 开发移动 App 的框架，Expo Go 可以让你在手机上快速预览开发中的 App。**）

接着，他让 ChatGPT 先生成项目计划和 PRD。

PRD 是产品需求文档。（**它会说明产品目标、核心功能、用户场景、界面要求和后续开发步骤，方便开发者或 AI 工具按计划执行。**）

ChatGPT 生成的方向很清楚：做“世界上最简单的计步 App”，没有干扰，没有复杂功能，只展示干净的步数。

这一步的价值在于：先让 AI 帮你把想法收敛成文档，再把文档交给 Cursor。

如果一开始只说“帮我做个走路 App”，AI 很容易发散。

## 用 Expo 把 App 跑起来

接下来，他在 Cursor 里创建项目。

ChatGPT 让他打开 terminal，运行 Expo 项目初始化命令，选择空白模板，然后启动本地服务。

这一步对程序员来说很普通，但对新手来说会很紧张。终端里不断滚动安装信息，很容易让人觉得自己做错了。

最后，Expo 生成了一个 QR code。

Gus 用手机里的 Expo Go 扫码，看到一个基础欢迎页面。电脑端也能打开 web 预览。

这一步看起来只是“跑通一个空壳”，但它非常关键。

它证明本地开发环境已经能工作。后面所有功能都可以在这个基础上迭代。

对新手来说，第一天能把 App 在手机上跑起来，就已经跨过了一个门槛。

## 把 PRD 交给 Cursor，让它做第一版界面

项目跑起来后，他把 ChatGPT 生成的 PRD 复制到 Cursor 里，告诉它：

请把这份文档作为基础，开始构建这个 App。

Cursor 读取项目后，开始修改文件，做出一个基础界面。

第一版界面只展示三个静态数字：今天、本周、本月。

但很快就报错了。

Gus 在 Expo Go 里看到 syntax error，也就是语法错误。（**语法错误是代码写法不符合语言规则，程序无法运行。**）

他不懂具体原因，于是把手机截图发给 AI。Cursor 判断是代码里数字写法有问题，然后修复。

这就是新手用 AI 编程的典型流程：

看到错误；

截图或复制错误；

交给 AI；

让 AI 修；

重新运行。

这个过程并不优雅，但能推进。

## 新手最容易卡住的地方：到底该信谁？

过程中还有一个很真实的问题：他一边用 ChatGPT，一边用 Cursor，不知道该听谁。

ChatGPT 会给一段代码，Cursor 也会自己修改项目。两个工具有时会给不同建议。

这会让新手很困惑：我是把 ChatGPT 的代码直接复制进去，还是让 Cursor 自己改？

这其实是 AI 编程早期很常见的问题。

更好的做法是把角色分清：

ChatGPT 更适合规划、解释、生成思路、判断下一步。

Cursor 更适合直接理解代码库、修改文件、运行项目。

如果两边都同时指挥代码，很容易混乱。

这一天 Gus 还在摸索，但他至少学到一点：不要想一次性理解所有文件夹和配置，先保持小步前进。

## 界面从能跑，变成勉强能看

修完错误后，界面终于能显示三个数字。

一开始数字太小，挤在屏幕中间，看起来不像真正的 App。

他让 Cursor 把数字放大，让整个界面更像一个极简仪表盘。

调整后，页面显示出三个大数字。虽然还不漂亮，但已经足够表达产品想法。

这时的 App 仍然没有真实计步数据。它只是静态数字。

但这没关系。

下一步，ChatGPT 建议先模拟动态数据，不要立刻接 Apple Health。

Apple Health 是苹果的健康数据系统。（**如果要读取真实步数，通常需要处理权限、健康数据接口和设备差异，对新手来说会更复杂。**）

于是 Gus 先让 Cursor 使用 React state 设置默认步数。

React state 是 React 里的状态机制。（**可以理解为界面上的动态数据，比如步数从 9300 变成 9400，页面会跟着更新。**）

然后再用定时器让数字每隔 3 秒自动增加。

这不是真实计步，但它能模拟用户看到步数变化的体验。

## 第一天的成果：一个会自动跳数字的原型

到最后，这个 App 可以显示今天、本周、本月三个步数数字，并且数字会每隔几秒增加。

它还没有接真实健康数据。

它还没有重置逻辑，比如每周一清零、每月第一天清零。

它还没有图标、品牌、注册、分享、付费。

但它已经不是一个想法。

它是一个可以在手机里打开的原型。

这就是一美元挑战第一天最重要的成果。

## 最后一步：公开构建

做完这个小原型后，Gus 没有只把它留在本地。

他去 Twitter 发了一条更新，大意是：

Day 0.01，开始构建下一个伟大的走路 App。之前那个版本先扔掉，从这里重新开始。

这一步也很重要。

公开构建不是等产品完美再发，而是在产品很粗糙时就开始记录。（**公开构建指持续分享从想法、开发、发布到失败或增长的全过程。**）

它会带来压力，但也会带来反馈和责任感。

对 Gus 这种新手来说，第一天最有价值的不是做出了多厉害的功能，而是完成了一个完整动作：

想法收敛；

生成 PRD；

创建项目；

跑通 Expo；

让 Cursor 做界面；

修错误；

模拟动态数据；

公开发布进展。

这就是从 0 到 0.01。

虽然离赚第一美元还远，但已经开始了。
