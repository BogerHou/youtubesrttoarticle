# 2026-04-23-Anthropic 的产品团队为什么能比别人跑得更快

如果只用一句话概括我在 Anthropic 做产品的感受，那就是：AI 产品的节奏已经完全变了。

过去，做一个功能可以按 6 到 12 个月规划。因为代码很贵，工程产能很有限，所以产品经理大量时间都花在路线图、跨团队对齐、依赖管理和发布节奏上。你要确保另一个团队的功能先上线，自己的功能才能继续往前走。

现在不一样了。随着模型能力提升，工程速度被大幅加速。很多产品功能的周期，已经从 6 个月变成 1 个月，有时变成 1 周，甚至变成 1 天。

这不是一句夸张的口号。它直接改变了产品经理的工作方式。

我在 Anthropic 负责 Claude Code 和 Cowork。Claude Code 是面向开发者的 AI 编程工具，Cowork 则更像是把 Claude Code 的 agent 能力带到非代码工作里。我的搭档 Boris Cherny 是团队技术负责人，也是很强的产品愿景设定者。他常常会把产品 3 到 6 个月后的样子讲清楚：如果模型继续变强，这个产品最终应该变成什么。

我的工作更多是回答另一个问题：我们今天站在这里，怎么走到那个愿景？中间要先做什么，哪些团队需要提前参与，营销、销售、财务、算力、文档、开发者关系这些环节有没有障碍，功能准备好以后能不能马上发出去。

很多时候，产品和工程之间的边界其实很模糊。我们大概 80% 的判断是高度一致的，剩下 20% 里，有些事情我更在意，我来推动；有些事情 Boris 更在意，他来推动。AI 原生产品就是这样，职位边界会变淡，真正重要的是谁能把问题往前推。

## AI 时代的产品经理，第一件事是缩短“想法到用户”的距离

我面试过很多产品经理。最常见的问题不是他们不聪明，而是他们还在用上一代软件公司的方式理解 AI 产品。

以前，一个产品经理很容易把重点放在多季度路线图、依赖协调、完整 PRD、正式发布流程上。这些事情仍然有价值，但在 AI 原生产品里，它们不是最核心的能力。

现在最重要的问题变成了：

我能不能让一个想法在一周内到用户手里？

如果工程师今天有一个想法，产品经理今天看到一个用户反馈，我们有没有办法在本周就做出一个可用版本，让真实用户开始试？如果没有，速度就会慢下来。

为了做到这一点，产品经理首先要把目标说得非常清楚。大语言模型（Large Language Model，LLM）本身太通用了，越通用越容易带来模糊性。你必须明确：我们到底为谁做？解决什么问题？最重要的使用场景是什么？

比如，在 Claude Code 里，如果我们要减少权限确认弹窗，目标不能只是“少弹一些提示”。更清楚的说法是：我们的核心用户是专业开发者，问题是他们在高频使用 Claude Code 时会出现权限提示疲劳，我们希望企业里的专业开发者能在安全前提下尽量接近零权限提示。

这个目标一旦说清楚，很多方案自然就被排除了。它既强调效率，也强调企业安全，还限定了用户群。团队不需要反复问产品经理“这个方向行不行”，因为原则已经摆在那里。

第二件事，是建立可重复的快速发布流程。

Claude Code 的很多功能都会以研究预览（research preview）的方式发布。我们会清楚告诉用户：这是早期产品，是一个正在验证的想法，我们想听反馈，也不承诺它永远保持现在的样子。这样做的好处是降低发布承诺，让团队敢于把东西更快拿出去。

第三件事，是让跨职能协作变得低摩擦。

我们有一个常设的发布频道。工程师如果觉得某个功能已经准备好，而且团队内部已经试用过，就会把它发到那里。文档负责人、产品营销、开发者关系同事会很快接上，通常第二天就能把文档和发布说明准备出来。

产品经理在这里的价值，不是拦在门口审批每件事，而是把这种机制搭好。好机制会让每个工程师都知道：我做完一个功能以后，下一步该找谁，别人会期待什么，我怎么把它顺利送到用户手里。

## PRD 没有消失，但它不再是默认动作

这并不意味着我们完全不写 PRD。

我们仍然会写，只是写得更克制。对于特别模糊的功能，一个一页纸的 PRD 很有帮助。它会说明目标、最让人惊喜的使用场景、现在有哪些失败模式需要修掉。

如果一个项目需要大量基础设施建设，周期确实会拉到几个月，那我们也会写更完整的 PRD。

但更多时候，我们靠两类东西让团队自己做决定。

第一类是指标。我们每周会和整个团队做一次指标复盘，让每个人都真正理解业务的各个方面：关键目标是什么，趋势如何，哪些因素在驱动变化。

第二类是团队原则。我们会明确写下核心用户是谁，为什么他们是核心用户，我们愿意做哪些取舍。这样，团队成员在日常决策里就不需要总是等产品经理拍板。

真正高效的 AI 产品团队，不能让所有判断都排队等一个人批准。

## 我们为什么能发得这么快

外界经常问，Anthropic 的发布速度是不是因为我们内部能用最强的前沿模型。

模型当然有帮助。我们能使用非常强的模型，这很幸运。但我不认为它解释了大部分速度提升。真正起作用的是流程和团队预期。

我们的流程很少。我们的目标是移除每一个影响发货的障碍，让团队里的每个人都相信：一个想法可以在不到一周的时间里从脑子里走到真实世界，有时甚至一天就够。

这背后也有代价。

我们牺牲了一部分产品一致性。过去，代码很贵，你会小心规划整个产品套件：每个产品服务哪个场景，它们怎么整合，用户应该走哪条路径。现在 AI 变化太快，要验证的想法太多，有些功能之间会出现重叠。

有时，这是故意的。我们内部喜欢两个不同形态，但不确定外部用户更喜欢哪一个，那就都拿出去试。

问题是，新用户会困惑：我到底该用哪个入口完成某件事？

所以我们后来做了 `/powerup` 这样的功能。它会带用户了解 Claude Code 的最佳实践。早期我们不想做内置引导，因为我们觉得产品应该足够直观，不需要教程。但功能越来越多以后，用户会问：这里有 100 个功能，我最应该先学哪 10 个？这时，内置上手流程就变得必要。

快速发布还会让用户觉得自己很难跟上。传统软件一个月或一个季度发一次大功能，用户隔一段时间看看更新就行。但 agentic tools（智能体工具，**指能自主规划、执行任务并根据反馈调整的 AI 工具**）每天都在变，很多人会觉得必须天天刷社交媒体，才不会错过新东西。

这也是我们需要解决的问题：工具本身应该帮助用户理解最新能力，而不是把用户推上越来越快的跑步机。

## AI 公司还需要产品经理吗？

很多人问，既然工程师现在能用 AI 写更多代码，产品经理是不是会变少？

我的观察正好相反：角色正在融合。

产品经理会做一些工程工作。工程师会做产品工作。设计师也会做产品判断，甚至直接落代码。你可以选择雇更多有产品品味的工程师，也可以在工程人数不变的情况下雇更多产品经理来帮助判断方向。

在 Claude Code 团队，我们非常重视有产品品味的工程师。团队里有很多工程师可以从用户在 Twitter 上的反馈出发，到周末前自己把功能做出来，几乎不需要产品经理介入。这是非常高效的工作方式。

但这不代表产品能力不重要。恰恰相反，当代码越来越便宜，最值钱的能力就变成：决定写什么。

什么样的用户体验是对的？用户第一次接触这个功能时，最自然、最惊喜的路径是什么？我们每天会看到大量 GitHub issue，用户会要求各种各样的东西。不是每个请求都值得做，也不是每个请求都应该按用户原话做。决定哪些值得做，以及怎么做，这就是产品品味。

工程背景在今天仍然很有用，因为它能帮助你判断一件事大概有多难。如果一个功能很容易，也许不用开会争论，一个小时做出来试试就行。如果一个功能很重，你就要提前知道它会占用团队多少资源。

但我不会预测这个优势会持续多久。AI 编程能力每隔几个月就会有明显变化，角色价值也会跟着变。更稳定的能力是第一性原理思考：看清技术环境怎么变，团队真正缺什么，然后低自尊地补上那个空缺。

这里的“低自尊”不是没有自信，而是不执着于“这是不是我的职责”。AI 时代的好产品经理要能戴很多帽子，也要能随时换帽子。

## 人类暂时还不可替代的地方

如果只看模型能力，未来很容易让人焦虑：人类的大脑还能在哪些地方有价值？

我现在看到的答案是三个。

第一，选择做什么。市场往哪里走，模型能力到哪里了，用户行为会怎么变，这些判断仍然需要人。

第二，判断做出来的东西是否真的好。模型可以给出很多方案，但最后哪个该进入产品，哪个该被删掉，仍然需要人做决策。

第三，是常识和情商。

任何一次产品发布背后都有上千个移动部件。模型不一定知道所有利益相关方是谁，他们之间有什么关系，各自偏好什么，应该在哪个场合沟通。很多 tacit knowledge（隐性知识，**指组织里没有写成文档、但靠经验和关系理解的知识**）现在仍然很关键。

这也是为什么我们很看重能在混乱里保持稳定的人。

在 Anthropic，事情经常一层叠一层。你可能周日晚上觉得某个问题是 P0，到了周一早上又出现一个新的 P0，周一下午又出现更大的 P0。过几天回头看，你会觉得自己周日担心的事情已经没那么重要了。

这种环境下，你必须睡好，才能第二天做正确决策。你必须非常残酷地排序，知道什么最重要，也要接受有些东西不会像你希望的那样完美。

有些功能上线时会有 bug。以前这会让我睡不着。现在我能接受，只要它没有阻塞核心用例，我们就能快速听到反馈，在下一个版本修掉。

## Anthropic 的一个底层优势：使命感让取舍更快

外界看到 Anthropic 的增长，常常会把它归因于模型强、产品强、市场时机好。这些都重要，但我认为还有两个更底层的东西。

第一个是统一使命。

我们招的人，通常都很在意把安全的通用人工智能（AGI，Artificial General Intelligence，**通常指具备广泛任务能力、接近或超过人类通用认知能力的 AI 系统**）带给全人类。这个使命不是挂在墙上的口号，而是会被拿来做实际取舍。

如果两个优先级冲突，我们会问：哪一个更符合 Anthropic 的使命？一旦做出决定，大家会站到同一个方向上。

第二个是组织层面的聚焦。

对我来说，使命和聚焦不完全一样。使命意味着一个团队愿意牺牲自己的目标和关键结果（OKR），去服务 Anthropic 整体目标。一个极端例子是，如果 Claude Code 失败了，但 Anthropic 成功了，我会非常高兴。团队也愿意沿着这个逻辑做决定。

这会让跨组织决策快很多。每个人不是只守着自己的产品线，而是先问整体目标是什么。

## 产品团队其实不大，但分工很清楚

Anthropic 的产品经理团队现在大约有 30 到 40 人。

研究产品团队由 Diane 负责，他们会理解客户对模型的反馈，把这些反馈传给最合适的研究团队，并推动模型发布。Claude Developer Platform 团队维护 Claude Code 所依赖的 API，也会发布 managed agents（托管智能体，**指由平台代为托管和运行的 agent 能力**）这样的能力。

Claude Code 团队负责 Claude Code 和 Cowork 的核心产品。企业团队负责让 Claude Code 和 Cowork 更容易被企业客户采用，工作包括成本控制、RBAC（Role-Based Access Control，基于角色的访问控制）、安全控制等。增长团队则负责整个产品套件的增长，也会和我们一起推动 Claude Code、Cowork 以及 Claude API 相关增长。

这种分工听起来传统，但日常工作并不传统。因为工程、产品、设计都在融合，我们更看重一个人能不能跨边界解决问题，而不是只守住自己的职能名称。

## 快速发货也会遇到事故，关键是把它当成流程问题

访谈里也聊到过 Claude Code 源码泄露事件。

我们看到以后立刻做了调查。结果是一次人为错误：有人在和 Claude 一起写一个 PR，内容只是更新包发布方式。这个 PR 实际上经过了两层人工 review，但问题还是发生了。

所以我们把它视为流程失败，而不是把责任简单推给某个人。这个人仍然在 Anthropic。更重要的是，我们已经加固了发布流程，增加更多防护，避免类似问题再次发生。

另一个外界关心的问题，是有些第三方工具过去会使用 Claude 订阅额度，后来我们做了限制。这个决定不轻松。Claude 需求很高，我们一直在扩展基础设施，也在让 harness 更省 token，让用户能用得更多。但订阅本来不是为第三方产品设计的，第三方产品的使用模式和第一方产品不同。

我们花了不少时间想怎么让过渡尽可能顺滑，也给订阅用户配套了一些额度。但最后仍然要做取舍：优先保障第一方产品和 API。这不是一个讨好所有人的决定，却是当时必须做的产品和资源选择。

## Claude Code、桌面端、移动端和 Cowork 到底怎么分工

很多人会混淆 Claude Code、Claude 桌面端、网页端/移动端和 Cowork。

我的理解很简单：如果输出是代码，我会用 Claude Code；如果输出不是代码，我会用 Cowork。

Claude Code 的命令行界面最适合一次性启动一个或几个编码任务。CLI 是我们最早的产品界面，也是很多新功能最先落地的地方，所以它通常是最强大的。

桌面端在前端工作里很有用。如果我在做一个 Web 应用，我会一边用 Claude Code，一边在桌面端右侧打开预览面板。这样我能实时看到页面效果，也能和 Claude 对话修改。

桌面端还适合不习惯终端的人。终端对非技术用户很陌生，会弹出一些吓人的权限提示，也不像普通软件那样能点来点去。桌面端更图形化，还能作为控制台，看见 CLI、桌面、网页和移动端启动的所有任务。

网页端和移动端解决的是另一个问题：人在路上怎么办？

很多人已经习惯让 agent 跑任务，但如果只能在本地电脑上运行，你就会看到有人在外面把笔记本打开，连着手机热点，只为了让 agent 别停下来。移动端的价值就是让你在散步、通勤、离开电脑时也能启动任务。

Cowork 负责的是非代码输出。比如清空 Slack、整理 Gmail、为客户会议做幻灯片、写功能目标文档、写发布计划。这些工作最终产物不是代码，所以 Cowork 更合适。

但 Cowork 要真正有用，第一步不是写一个神奇提示词，而是接入上下文。我的 Cowork 会连接 Google Calendar、Slack、Gmail、Google Drive。你应该连接自己的沟通工具，以及团队事实来源所在的地方。只有拿到足够上下文，它才能给你产出真正贴合工作的结果。

## 一个真实案例：让 Cowork 先做 20 页幻灯片草稿

我最近要在 Code with Claude 大会上做一次分享，主题是 Claude Code 如何从一个助手变成完整的 agent。我希望展示我们发布过哪些产品，怎样推动这个转变，也想找一些内部成功案例作为 demo。

我们的产品营销同事 Alex 已经写了一版建议。我把这些材料、我想讲的叙事、我不满意的旧草稿、我手工做过但也不满意的版本，都交给了 Cowork。

我大概给它的任务是：帮我为 Code with Claude 大会做一套幻灯片。这是产品营销建议覆盖的内容。这是我现在不喜欢的草稿。这是我手工做的另一个版本，也不满意。先生成一个带细节的提纲。同时注意，不要和更重要的 keynote 内容重复太多。

Cowork 会先读我给的链接，生成一个提纲和若干可能覆盖的点。我再决定最终应该讲什么。

我的最后选择是：从让本地任务成功，到让每个 PR 变绿，再到帮助工程师合入更多 PR。每一段都要配最有说服力的 demo。

我做完这个决策后，Cowork 会继续工作。它会看 Twitter 上我们发过什么，查常设发布频道，读 Claude Code announce 频道里团队发的 demo，再把这些材料综合成一份大约 20 页的幻灯片草稿。

第二天早上我读到这份草稿时，它已经相当不错。它有点太啰嗦，因为我喜欢文字极少的幻灯片，所以还需要一轮反馈。但它已经把最耗时的搜集、组织和初稿工作做掉了。

更重要的是，因为我给了它我们对外演讲常用的标准模板，它能学到颜色、字体、版式和幻灯片格式，所以视觉上看起来像 Anthropic 设计师做的。

这就是今天产品经理仍然重要的地方：Claude 是很好的头脑风暴伙伴，能快速综合大量信息，也能展示所有可能性。但最后哪些内容应该进入产品，哪些 demo 最有说服力，叙事应该怎么取舍，仍然由人决定。

## 内部工具会越来越个人化

Claude Code 还带来了一个变化：公司内部出现了很多个性化工作软件。

过去，如果现成 SaaS 不能完全适配某个工作流，大家只能忍着用。现在，做一个定制小工具的门槛变低了。

例如，Claude Code 销售团队里有同事发现自己总是在重复做客户幻灯片。他于是做了一个 Web 应用，里面有几套我们知道效果很好的 Claude Code 核心演示材料，比如入门版、进阶版、精通版。

然后，这个工具能接入客户上下文，从 Salesforce、Gong 和会议笔记里拉信息。它会判断客户用的是 Bedrock、Vertex、Claude for Enterprise 还是 Console，因为这会影响可用功能。它会识别客户是不是关心软件开发生命周期（SDLC，Software Development Life Cycle）里的代码审查阶段，就自动加入代码审查功能页。它还会看客户是否需要 HIPAA 合规或特定安全控制，于是补上相应幻灯片。

如果客户使用 Vertex 或 Bedrock，并且不想用 Claude for Enterprise，那工具就会移除只适用于 Claude for Enterprise 的页面。

以前，这种定制可能要 20 到 30 分钟。很多人要么花时间做，要么干脆用通用版本。现在几秒钟就能拿到一份更贴合客户的材料。

这类工具不是为了炫技，而是直接减少重复劳动。

## 除了工程，最重度使用 AI 的是 Applied AI 团队

如果只看 token 消耗，工程团队当然很重。但在工程之外，Applied AI 团队也非常重度使用 Claude Code 和 Cowork。

Applied AI 可以理解成非常技术化的 go-to-market 团队，类似 forward deployed engineering（前线部署工程，**指深入客户场景、帮助客户把技术真正落地的一类技术岗位**）。他们会帮助客户采用最新 API 和模型能力，既用于客户自己的产品，也用于客户内部提效。

这个团队经常要同时覆盖多个客户。一天多的时候，一个人可能有 5 到 10 场客户沟通。前一天晚上，他们会让 Cowork 整理第二天所有会议：客户最近问过什么，最关心什么，上次会议留下哪些行动项，哪些功能 ETA 需要确认。

Cowork 还能去 Slack 里找最新信息，把它整理进会议 brief。这样 Applied AI 同事进入客户电话时，手里就是最新上下文。

随着模型变强，人们会把更多任务交给它，花在 Claude Code 和 Cowork 这类工具里的时间也会增加。每次模型跃迁或产品能力明显提升，单个工程师或知识工作者的 token 成本都会上升。现在它仍然低于平均工程师薪资，但占比在提高。

在公司内部，我们可以使用很多 token，但不是无限浪费。团队很清楚服务这些模型有多贵，所以浪费 token 是非常不被鼓励的。我们相信每个人能做负责任的判断。

## 未来产品经理最稀缺的能力：刚刚好的 AGI 信念

如果问我今天 AI 公司最看重产品经理什么能力，我会说：能定义一个月后的产品应该是什么样。

这很难。因为模型一个月后能做什么，用户行为一个月后会怎么变，都有很大不确定性。最好的产品经理能从用户如何“滥用”现有产品边界里看到模式，然后设定方向，持续执行。如果模型能力比预期更强或更弱，再调整路径。

这里有一个很微妙的能力：你要有刚刚好的 AGI 信念。

如果你完全不相信模型会变强，你会做出过时的产品。如果你太相信未来的超强模型，你又会设计得过于简单。

为超级 AGI 设计产品其实很容易。假设模型什么都会，那你只需要一个文本框。用户说一句话，它知道需要什么工具、什么集成、什么时候该追问、什么时候自己执行。

真正难的是：面对当前模型，怎么最大化激发它的能力？怎么把用户引导到黄金路径（golden path，**指最容易成功、最推荐的标准使用路径**）？怎么让用户利用模型强项，同时补上模型弱点？

这个能力很稀缺。

培养它的方法不是只看报告，而是大量使用模型。你要花很多时间和模型对话，也要让模型反思自己的行为。

比如，有时模型改了前端代码，也跑了测试，但没有真的打开 UI 看一眼。我会问它：你为什么没有做 UI 验证？它可能会说，系统提示里有某个地方让它困惑；也可能说它没意识到前端验证属于这个任务；还可能说它把验证交给了子 agent，但没有检查子 agent 的工作。

这些回答不一定每次都完全可靠，但它们能帮你发现 harness（运行框架，**指包在模型外面、负责工具调用、提示词、权限和任务流程的系统**）哪里误导了模型。你就可以改系统提示、改工具、改流程，把失败补上。

第二个方法，是找到你信任的少数高质量用户。很多人都会反馈，但不是所有反馈都同样有判断力。通常会有五个左右的人，特别擅长说清楚某个模型和某个产品框架组合为什么好、哪里不对。找到他们，能让你更快得到有用反馈。

第三个方法，是做评测（evals）。

评测不需要一上来就做几百个。十个高质量 eval 就很有用。它能把“成功是什么”具体化，让团队知道目标、进度和缺口。某些功能，比如记忆（memory），尤其适合用 eval 来定义。

我个人不是每天都写 eval，但如果某个功能需要更多产品定义，我会直接做几个：这些是测试用例，这是运行方式，哪些成功，哪些失败，这是我用来提高成功率的提示词。

产品经理写 eval，其实是在把品味变成可检验的标准。

## 模型变强以后，产品要学会删东西

AI 产品很特别的一点是：新模型上线后，你经常要回头重新审视已经做过的功能。

很多时候，我们不是增加功能，而是删除不再需要的拐杖。

一个典型例子是待办清单。Claude Code 早期刚发布时，用户会让它做大型重构。它会说：“好的，我需要改 20 个调用点。”然后它改了 5 个就停了。

我们开始想，怎样让它记得把 20 个都改完？团队里的 Sid 想到，真实开发者会怎么做？人在 VS Code 里会搜索所有调用点，左边出现一个列表，然后一个个处理。那我们能不能给 Claude 一个类似工具？

于是他加了待办清单。效果很好，Claude 能更完整地处理所有调用点。

但到了 Opus 4 以及后来的模型，我们发现不再需要强制它使用待办清单。早期模型需要我们不断提醒：“你完成待办清单了吗？没完成不能结束。”新模型会自然地想到要用清单，也会自然地把清单做完。

现在，待办清单更多是给用户看的。它能让人更清楚 Claude 正在做什么，但对模型完成任务已经不是必需。

每次发布新模型，我们都会读完整个系统提示，逐段问自己：模型还需要这个提醒吗？如果不需要，就删掉。

更令人兴奋的是，新模型会让以前做不成的功能突然变得可行。

代码审查就是例子。我们尝试过几次做代码审查产品，也发过更简单的 `/code review` 命令。但只有在最近一代模型能力足够好以后，我们才觉得：现在它真的可以成为工程团队合入 PR 前依赖的审查工具。

新的模型让我们可以同时运行多个代码审查 agent，遍历整个代码库，再综合出工程师合入前需要处理的真实问题。

这说明，AI 产品有时要提前做那些“现在还差一点”的原型。你要知道缺口在哪里。等新模型出来，把它换进去，就能立刻验证能力是否补上了。

## 长期愿景：从一个任务，到同时管理上百个 Claude

我们思考 Claude Code 和 Cowork 的长期方向时，会从构建块开始。

第一个构建块是单个任务成功。用户给出清楚提示，模型能不能稳定产出可接受结果？这个结果可能是一段可以合并的代码，也可能是一份可以分享给同事或客户的文档。

当单个任务成功率提高以后，用户会自然开始同时跑多个任务。2025 年末，多任务编码变得很常见。一个任务可行以后，用户会想同时跑 6 个。

再往后，可能就是同时跑 50 个 Claude，甚至上百个 Claude。

那时，你大概率不会把所有任务都跑在本地机器上，因为内存不够。任务会更多地远程运行。新的问题会变成：人类怎么管理这些任务？界面怎么告诉你哪些任务需要看？agent 怎么充分验证自己的工作？当它说“完成了”，你怎么快速确认并相信它确实符合你的要求？

还有一个关键点：整个过程要能自我改进。如果某次任务结果不符合你的偏好，你给了反馈，未来所有类似任务都应该吸收这个反馈，不要再犯同样的错。

这就是我们想把用户带向的方向。

## 给普通职场人的建议：别只玩 AI，要让它每天替你干活

AI 给每个人都增加了杠杆。我的建议很直接：每当你发现自己在重复做某个手工任务，就想想能不能用 Claude Code、Cowork 或其他 AI 工具自动化。

大多数人的工作里都有两部分：一部分是创造性的、你真正喜欢的；另一部分是重复、琐碎、你不想做但必须做的。

AI 最适合先接手后一部分。它可以从你过去如何做这些任务中学习，泛化成流程，再自动执行。这样你就能把时间放回更有创造力的事情上。

但这里有个陷阱：不要把一个 90% 或 95% 准确的流程误以为是自动化。

如果一个自动化不能稳定工作，它就不是真正的自动化。最后 5% 到 10% 往往最费时间，但也最重要。你需要投入精力教它你的偏好，给它反馈，让它更新技能，直到你真的能依赖它。

我自己也踩过这个坑。我一直在教 Cowork 帮我把 Gmail 做到 inbox zero，但这件事很耗时，还没有完全成功。产品上我们也需要把这种自定义流程做得更顺，让用户不用理解太多概念，比如什么是 skill、怎么给 skill 反馈、怎么让 Cowork 根据反馈更新 skill、怎么检查更新后的内容。

还有一个建议：不要只做一次性原型。

很多人会用 AI 一次性生成一个小应用，看起来很酷，然后再也不用。这能证明工具强，但不能给你带来真正杠杆。更有价值的是做你每天都会用的应用，或者自动化你每天都会遇到的工作。

另一个极端也要避免：过度沉迷于定制工具链。有人会花大量时间加技能、加 MCP（Model Context Protocol，**让模型连接外部工具和数据源的一种协议**）、调工作流、晒配置，但忘了自己原本要交付的产品或功能。

我越来越相信，简单配置往往更有效。

2024 年那代 AI 产品主要是聊天式的。Claude Code 这一代产品更像行动式的。真正的“啊哈时刻”不是模型告诉你该怎么做，而是它真的替你做了。

所以，如果你想在 AI 时代过得更好，不要只围观，不要只收藏技巧。找一个自己反复遇到的问题，把它自动化到足够可靠。然后把省下来的时间，用来做团队一直想做、但过去没人有带宽做的事情。

我的工作信条很简单：Just do things，直接做事。

如果你理解目标，理解约束，也能用第一性原理推导出正确行动，那就别等太久。快速做，做错了就学习、修正、道歉，然后继续往前。

在 AI 产品时代，最稀缺的人不是职位描述最漂亮的人，而是能在混乱里看清目标、补上缺口、把事情送到用户手里的人。
