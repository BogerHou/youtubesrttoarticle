# 2025-12-14-OpenAI内部：2026为什么会是智能体真正上场的一年

我以前听到“AI 编程”这几个字，第一反应通常是：它能不能帮我把代码写快一点？

但这次听 OpenAI Codex 产品负责人 Alexander Imbiricos 的分享，我发现 OpenAI 内部想的已经不是“更快写代码”这么简单了。他们真正想做的是：让 Codex 从一个会写代码的工具，变成一个能参与计划、实现、验证、部署、维护的软件工程队友。

这也是为什么他说，AI 最大的瓶颈不一定是算力。一个更容易被低估的瓶颈，是人的输入速度、人的多任务能力，以及人类检查 AI 工作的速度。

换句话说，如果 AI 每天本来可以帮我们一千次，但我们只会主动提示它几十次，那真正卡住生产力的，可能不是模型不够强，而是它还没有真正融入我们的工作流。

## 在 OpenAI，速度和野心被重新定义

Alexander 加入 OpenAI 大约一年。在那之前，他做了五年创业公司，更早之前在 Dropbox 做产品经理。

他原本也觉得创业公司已经够快了。每个创始人都会说，自己的团队人才密度高、决策快、野心大。但到了 OpenAI 之后，他说自己对“快”和“野心”的理解被重塑了。

最直观的例子就是 Codex 本身。

从 8 月 GPT-5 发布之后，Codex 的增长非常爆炸。OpenAI 上一次对外提到时，Codex 的规模已经比 8 月高出 10 倍以上；而在这次访谈里，Alexander 说这个数字已经到了 20 倍。Codex 模型现在每周服务“很多很多万亿”词元（token，模型处理文本的基本计量单位），也基本成了 OpenAI API 中调用量最大的编程模型。

但更有意思的不是数字，而是 OpenAI 做产品的方式。

传统产品管理很强调“先瞄准，再加速”。先把方向定清楚，再组织团队往前冲。可在 AI 领域，很多能力本身还在快速变化，你甚至不知道下个月模型会出现什么新能力，也不知道某个能力在技术上可行之后，用户是否真的会用。

所以 OpenAI 更像是在“模糊瞄准，快速试错”。

他们会对一年以上的未来做很认真、很深入的讨论，也会对未来几周或几个月要做什么做很具体的判断。但中间那段最尴尬的时间跨度，反而最难规划。因为它太远，不够确定；又太近，不能只靠愿景。

这就形成了一种非常自下而上的组织方式。团队需要快速把东西做出来，让真实使用来告诉他们什么有效。

Alexander 也提醒，这套方式不是任何公司都能照搬的。因为它依赖非常高的人才密度、个人驱动力和自主性。如果团队没有足够强的判断力，只说“我们也要自下而上”，结果可能只是失控。

## Codex 不是一个插件，而是一个“软件工程队友”

如果只从今天的形态看，Codex 是 OpenAI 的编码智能体（coding agent）。

你可以把它装成 VS Code 这样的集成开发环境（IDE）扩展，也可以在终端里使用。它能回答关于代码的问题，写代码，运行测试，执行命令，帮你完成软件开发生命周期中间那一大段工作：把代码写出来，并让它最终能进入生产环境。

但 OpenAI 对 Codex 的定义更大。

他们认为，今天的 Codex 只是“软件工程队友”的开端。真正的队友不只是接到任务后写一段代码。它应该能参与更早的想法和计划阶段，也能参与更后面的验证、部署和维护阶段。

Alexander 用了一个很形象的比喻：今天的 Codex 有点像一个很聪明的实习生，但它不会主动读 Slack，也不会主动看 DataDog 或 Sentry，除非你叫它去看。

DataDog 和 Sentry 都是工程团队常用的监控、告警和错误追踪工具。（**也就是说，一个真正可靠的工程队友不只会写代码，还要知道线上系统有没有出问题、用户反馈在哪里、团队最近在讨论什么。**）

所以现在大多数人还是在和 Codex “结对”。你告诉它要做什么，它帮你做。可 OpenAI 想要的下一步，是让它变得更主动。

一个好队友不会永远等你下命令。刚入职时，你会跟他开几次会，告诉他哪些沟通方式有效、哪些任务优先。再往后，你不会每件事都拆成指令，而是会说：你负责这块代码，和这些人协作，如果看到什么该做的事，也可以告诉我。

这就是 Codex 的核心目标之一：主动性（proactivity）。

Alexander 说，现在的 AI 产品其实很难用。不是因为模型不聪明，而是因为你必须时时刻刻想：“这件事能不能让 AI 帮我？”如果你没有提示模型，它大概率就不会帮你。

普通用户今天每天可能提示 AI 几十次，但一个真正聪明的实体，本来可能在一天里帮你上千次。Codex 要解决的问题，就是“什么样的智能体，能默认就是有用的”。

## Codex 增长的关键：先别离用户太远

Codex 早期有一个形态叫 Codex Cloud。

它在云端有自己的电脑，你可以把任务委派给它，它可以并行跑很多很多任务。这听起来很未来，也确实代表了 OpenAI 想去的方向：你把工作交给它，它异步完成，回来给你结果。

但问题是，这个未来形态对普通用户来说有点难上手。

你要配置环境，要给模型足够的工具来验证修改，还要学会怎么用异步方式提示它。Alexander 的比喻是：这就像你新招了一个队友，但永远不能和他开会，只能异步留言。对于一些人可以，但对大多数人来说，初期建立信任很难。

于是 Codex 团队往回走了一步。

今天，大多数用户第一次接触 Codex，是通过 IDE 扩展或命令行工具（CLI）。它直接在你的电脑上交互式工作，能看到本地依赖，也能在沙箱（sandbox）里运行命令。（**沙箱可以理解为一个受限制的执行环境，让程序可以运行，但不会随便破坏你的系统。**）

如果某个命令在沙箱里不能运行，它会来问你。这样用户不需要先搭建一套复杂的云端环境，就能马上进入反馈循环。

这件事也暴露了 OpenAI 内部“吃自己狗粮”（dogfooding）的局限。

OpenAI 内部的人长期训练和使用推理模型，很习惯先想清楚任务、并行发出去、过一段时间再回来检查。这种使用方式对他们很自然。但普通市场不一定这样工作。于是团队意识到：内部狗粮很重要，但它不能完全代表外部用户。

真正的增长来自一个更朴素的产品判断：先让用户在自己熟悉的工作方式里立刻获得价值，再逐步把他们带到更未来的委派式工作流。

## 为什么“任何智能体都可能需要写代码”

这次访谈里我最受启发的一段，是 Alexander 对智能体的判断。

如果我们想做一个超级助手，它必须能“做事”。不是只聊天，不是只回答问题，而是能影响你的真实世界。

过去一年 OpenAI 学到的一件事是：模型要做事，最好能使用电脑。而模型使用电脑有很多方式，比如点击界面、调用系统接口、读取屏幕。但 Alexander 认为，很多时候最有效的方式其实是写代码。

因为代码是可组合的、可复用的、可互操作的。

如果一个智能体只是坐在电脑前点击按钮，它每次都要重新摸索。但如果它能写脚本，能调用库，能复用团队已有的代码，那么它就不只是“模拟一个人操作电脑”，而是在真正使用计算机的能力。

这也是他提出的一个很强的判断：如果你想构建任何智能体，也许你本质上都应该在构建一个编码智能体。

对非技术用户来说，他未必知道自己背后用了一个 coding agent。就像普通人不会关心自己是不是在“使用互联网”，只会关心 Wi-Fi 开没开、事情能不能完成。

所以 Codex 表面上是软件工程队友，但更深一层，它在训练一种通用能力：让智能体通过写代码来使用电脑。

这也解释了为什么今天非编程领域的智能体还显得很早期。Alexander 的看法是，编码智能体已经相当有用，但编码之外的智能体还很早。等它们能以可组合的方式使用代码，能力会明显提升。

## 真正的新瓶颈：不是写代码，而是验证代码

当 AI 能写越来越多代码，软件工程师的瓶颈会发生变化。

过去，写代码是慢的。现在，写代码本身变快了，甚至快到一个团队可能同时冒出很多个 pull request。问题变成了：谁来检查这些代码？谁来确认它真的能跑？谁来承担“合进去以后别把生产环境搞坏”的责任？

Alexander 说，很多工程师最享受的其实是写代码，是进入心流、设计结构、把东西搭起来。可当 AI 写了大量代码之后，人类反而被推到了一个不那么有趣的位置：审查 AI 写的代码。

这就是 Codex 团队现在很关注的方向：不仅要让 AI 写代码，还要让 AI 更会验证自己的工作，也要让人更容易审查它的工作。

比如在 Codex Web 里，如果一个智能体做了某个界面修改，你第一眼应该看到什么？是 diff，还是它生成出来的界面预览？

如果目标是“最大化加速人类”，很多时候你应该先看结果图，而不是先陷进代码差异里。只有当 AI 已经做过初步审查，或者你需要深入确认时，人才应该进入代码层面。

这就是产品细节里的价值观：不是让人变成 AI 的质检员，而是让人仍然感觉自己被增强、被赋能。

## Codex 为什么突然变强

模型本身当然在进步。

Alexander 提到，OpenAI 刚发布了 GPT-5.1 Codex Max。相较 GPT-5.1 Codex，它在同类任务上的完成速度大约快 30%，在更高推理档位下也更聪明，尤其适合处理那些很难定位的棘手 bug。

他还提到 Andrej Karpathy 对 Codex 的评价：有些最难的 bug，他自己会花几个小时也找不到头绪，其他工具也解决不了；交给 Codex 跑一个小时，反而能解决。

但 Codex 的进步不只是“训练一个更强模型”。

OpenAI 越来越把智能体看成一个完整系统。这个系统至少有三层：模型、API，以及运行框架（harness，负责把模型接到工具、命令、文件和任务环境上的那层系统）。

一个例子是压缩（compaction）。

Codex Max 可以连续工作很久。有人会让它跑一整晚，甚至跑 24 小时。但模型长时间工作会遇到上下文窗口（context window）的限制。（**上下文窗口就是模型一次能“记住”和处理的信息范围，超出后就需要整理、摘要、转移。**）

为了解决这个问题，Codex 有了 compaction：当任务接近上下文上限时，模型知道自己可能要准备进入新的上下文；API 要理解这个过程；harness 要能准备好相关数据，把任务延续下去。

这说明 Codex 的竞争力不只是模型分数，而是模型、API、工具、沙箱、终端工作方式一起优化出来的结果。

Codex 团队选择了一条很明确的路线：让模型使用 shell，在终端里工作，再通过沙箱保证安全。这和一些依赖语义搜索、专用工具调用或复杂图形界面的方案不同。统一优化一个世界，速度会更快。

## 从 PM 到设计师，OpenAI 内部已经在改变工作方式

Codex 对工程师的影响很明显。但 Alexander 说，作为产品经理，他也感觉自己被放大了很多。

他引用了 Scott Belsky 的一个说法：压缩人才栈（compressing the talent stack）。意思是，当每个人都能借助 AI 做更多事时，角色之间的边界会变薄，团队需要跨越的沟通边界会减少，效率会提高。

在 OpenAI，PM 可以直接让 Codex 帮忙理解代码变化、回答技术问题、做原型。很多时候，做一个原型比写一份长规格文档更快。

他们还看到大量“用完即弃”的代码。比如有人想理解一批数据，以前可能懒得专门写工具，现在可以直接把数据交给 Codex，让它做一个交互式数据查看器。

设计团队也在被改变。

有一次，设计师想做 Codex 里的金币动画。正常情况下，专门编程实现一个动画编辑器太麻烦。但他们直接用 vibe coding 做了一个动画编辑器，再用这个编辑器做动画，最后把结果合进代码仓库。

他们甚至有一个用 AI 快速写出来的 Codex App 侧边原型。团队快速讨论之后，设计师不只是画图，而是直接做出可玩的原型。如果大家觉得方向对，再把这个原型推进成真正的 PR。Codex 的 UI 是 Rust 写的，所以有些部分设计师可能自己落地，有些则交给工程师帮忙收尾。

这不是“设计师替代工程师”，而是产品讨论的粒度变了。以前大家围绕文档和稿子争论，现在可以围绕一个会动的东西讨论。

## Sora 和 Atlas：OpenAI 内部加速的两个样本

访谈里最夸张的例子，是 Sora Android 应用。

这是一款全新的应用。团队从零开始，用 18 天做到了员工可用；再过 10 天，也就是总共 28 天，就公开发布。Alexander 说，这个过程是在 Codex 帮助下完成的，速度非常疯狂。

更关键的是，这不是一个庞大团队。他确认大概只有两三个工程师。

这里也有一个 Codex 特别擅长的场景：跨平台迁移。因为团队已经有 iOS 应用，底层 API 和系统很多已经想清楚了。工程师可以让 Codex 同时查看 iOS 和 Android，先生成工作计划，再把功能迁移过去。

所以它不是凭空发明一切，而是把已有系统里的模式识别出来，再搬到另一个平台上。

另一个例子是 Atlas。

Atlas 是 OpenAI 推出的浏览器。浏览器本身是很重的工程项目，不是几天就能“随便写出来”的小应用。但 Atlas 团队已经有很多 Codex 重度用户。有工程师告诉 Alexander，以前某些工作可能需要两三个工程师做两三周，现在变成一个工程师做一周。

Atlas 先发布了 Mac 版本，团队后来也在推进 Windows 版本。这个过程反过来帮助 Codex 在 Windows 上变得更好。Alexander 还提到，OpenAI 上周发布的模型，是第一个原生理解 PowerShell 的模型。PowerShell 是 Windows 的原生命令行语言。

这就是内部飞轮：团队用 Codex 加速真实产品，真实产品暴露 Codex 的缺口，Codex 再变强。

## 为什么 OpenAI 要做浏览器 Atlas

听到这里，Atlas 这件事也更容易理解了。

Alexander 以前做的是一个屏幕共享、结对编程相关的创业公司。他加入 OpenAI 后，一开始想做的是“上下文桌面助手”。因为他觉得，给助手喂上下文太烦了。

如果一个 AI 要帮我，我不想每次都解释我在看什么、我想做什么、这个页面为什么重要。最好的情况是，它已经理解我正在做的事，然后在正确的时刻给我正确的帮助。

而大量工作发生在浏览器里。

如果 OpenAI 自己做浏览器，就不需要依赖截图，也不用绕过各种桌面软件不稳定的无障碍接口。它可以更接近渲染引擎，直接理解网页内容，并在网页里做第一方交互。

Alexander 用游戏里的“上下文动作”（contextual action）来解释这件事。（**在很多游戏里，你走到一个物体前按一个键，系统会根据场景自动判断你是开门、捡东西还是互动。这个动作不是固定命令，而是由上下文决定。**）

未来如果 AI 每天帮你上千次，绝不能靠上千条推送通知来告诉你“我刚刚帮你做了什么”。那会很烦。

更好的方式是：当你正在看一个指标面板，发现某个核心指标下降时，AI 正好在旁边给出判断：它为什么下降，可能应该怎么修。

这就是混合主动权（mixed initiative）：人仍然掌控方向，但 AI 会在合适的上下文里主动浮现帮助。

他还用 Tesla 自动驾驶做类比。Tesla 让车自己开，但人仍然可以加速、调速度、微调方向，而且不一定要立刻退出自动驾驶。Alexander 认为，这是构建智能体产品的一个好范例：AI 给你很多帮助，但你仍然感觉自己在控制。

## 构建变容易后，什么会变得更重要？

当写代码变快，很多人会自然想到：那是不是创意更值钱了？是不是只要有点子就行？

Alexander 的答案很克制：点子仍然没有很多人想象得那么值钱。执行依然很难。

你可以更快地做出东西，但它仍然要是一个连贯的产品，仍然要能解决真实问题，仍然要能进入市场。分发很重要，对客户问题的深刻理解也更重要。

如果今天只能选择一种核心能力，他会选择：对某类客户的真实问题有非常有意义的理解。

所以他看好垂直 AI 创业。不是做一个什么都能做的泛用工具，而是非常深地理解一个具体场景，比如演示文稿、财务分析、客户支持、某个行业流程，然后把 AI 接入它的真实工作流里。

当“构建”本身变便宜，真正稀缺的就更像是：你知道该为谁构建，为什么构建，以及怎样让它真的被采用。

## Codex 团队怎么看进展

OpenAI 当然会看评测和基准测试。但 Alexander 说，像 Codex 这样的工具，很容易让团队沉迷于高级用户的深层功能。

所以他们会特别盯早期体验。

比如第 7 天留存（D7 retention，用户注册后第 7 天是否还回来使用）。他们也会重新从零注册、从新用户视角体验产品。Alexander 甚至开了不止一个 ChatGPT Pro 账号来做更真实的测试，每个月被收 200 美元，还开玩笑说应该去报销。

另一个重要信号来自社交媒体。

Codex 团队里有几个人几乎一直在看 Reddit 和 X。X 上更容易看到兴奋和传播，Reddit 上则更负面，但更真实。尤其 r/Codex 这样的社区，里面的抱怨和赞美都会被认真看。

这点很有意思。越是前沿产品，越不能只看仪表盘。因为一个 coding agent 可以用于太多场景，它可能在整体指标上不错，但在某个具体行为上很糟。真实用户的抱怨，往往比平均分更能暴露问题。

## 普通人应该怎么试 Codex

如果你第一次尝试 Codex，Alexander 的建议反而不是从小玩具开始。

他建议给它一个真正困难的任务。

比如你有一个很难定位的 bug，不知道原因是什么，那就让 Codex 帮你调查，甚至实现修复。Codex 的定位是专业工具，不只是让你随便 vibe code 一个小页面来判断好不好玩。

当然，“最难任务”也不是让它从零帮你建立一家独角兽公司。更合适的是一个清晰的问题、一个真实任务。

语言支持方面，Codex 的训练分布大致跟现实世界里的编程语言频率一致。除非你用的是非常冷门的语言，或者公司内部私有语言，否则它应该都能处理。

他的上手建议是：并行试几件事。

可以让它理解代码库，可以让它围绕一个想法和你一起写计划，也可以让它处理一个具体 bug。关键是把它当成一个新队友来建立信任。你不会把一个刚入职的人拉来就丢给他零上下文任务。你会先确认他理解代码库，再对齐计划，再让他一点点做。

同样，Codex 也需要这种协作方式。

## 年轻工程师还要不要学编程？

AI 越会写代码，很多人越焦虑：那我还要不要学计算机？还要不要学编程？

Alexander 的回答可以拆成三层。

第一，做事。

随着 coding agent 变强，一个大学生或刚毕业的人能做的事情，比过去多很多。对早期职业候选人来说，他会特别关注：这个人能不能用最新工具变得非常高产。因为 AI 缩小了新人和资深工程师之间的一部分差距。

第二，理解好系统。

未来也许不需要每个人都手写某个具体算法，或者记住每种循环怎么写。但理解什么是好的软件系统，理解系统工程，理解团队协作，仍然很重要。

AI 不会突然在没有人类帮助的情况下构建完美系统。更可能出现的是一个渐进过程：人类不断把智能体配置得更会验证、更会构建、更会看结果。

第三，站在知识前沿。

如果你在某个领域真的深入到前沿，智能体暂时不会比你更懂。更重要的是，当你推动前沿时，你会自然被迫使用 coding agent 来加速自己的工作流。

OpenAI 内部已经出现了这种情况。Codex 会写很多管理自己训练运行的代码，也会参与关键基础设施。Codex 的代码审查已经抓到过一些有意思的配置错误。

他们甚至开始探索让 Codex 为自己的训练“值班”。训练运行期间，会有大量图表需要人看，内部把这叫 babysitting。（**因为模型训练很贵，也很复杂，训练过程中的异常需要及时发现，有时要暂停、修复或调整系统。**）

未来的想法是，让 Codex 循环观察这些图表变化，判断训练是否异常，并帮助团队更高效地训练模型。

这已经超出了“帮人写代码”的范畴。它更像是一个系统开始参与维护自身。

## AGI 的瓶颈，可能是人的输入速度

最后，主持人问了一个绕不开的问题：离通用人工智能（AGI，Artificial General Intelligence）还有多远？

Alexander 没有给一个具体年份。他更关心的是生产力曲线什么时候出现“曲棍球杆式增长”。

他认为，现在一个被低估的限制因素，是人类写提示词的速度，以及人类多任务处理的速度。

你可以让一个智能体观察很多工作，但如果它不能验证自己的工作，人类还是会卡在审查代码、确认结果这一步。你让它生成 100 个 PR，最后仍然需要人一个个看，那生产力提升就会被验证环节吃掉。

所以真正要解锁的，是把这些生产力循环从“人类必须不断提示”和“人类必须手动评估”里解放出来。

他预计，从明年开始，早期采用者会先看到明显的生产力跃迁。比如一家创业公司要做一个新应用，如果它从一开始就选择更适合智能体工作的技术栈，可能会让 agent 更自给自足。

但像 SAP 这样复杂的大公司和老系统，不可能一夜之间让智能体端到端接管。它们需要慢慢替换系统、更新系统、配置工具，让智能体能处理越来越多工作。

所以这不会是一个二元开关。不是某天突然 AGI 来了，全世界同时改变。更可能是：新公司先出现生产力曲线陡升，几年后更大的公司陆续跟上。当这种生产力提升反过来流入 AI 实验室本身，继续加速模型和系统研发时，我们才会接近他所说的 AGI 层级。

## 2026 的关键词不是“聊天”，而是“做事”

听完整场访谈，我对“2026 是智能体之年”这句话有了更具体的理解。

它不是说我们会突然拥有一个无所不能的机器人，也不是说聊天界面会消失。聊天仍然是一个很好的通用入口，因为当你不知道该怎么使用 AI 时，聊天最自然。

但真正的变化，是 AI 开始离开单纯的聊天框，进入真实工作现场。

在代码里，它能写、能跑、能测、能查 bug。  
在浏览器里，它能理解你正在看的页面，并在合适时机给出上下文帮助。  
在团队里，它能从 Slack、监控、错误报告、用户反馈里发现该做的事。  
在产品开发里，它能让 PM、设计师、工程师之间的边界变薄，让原型比文档更快出现。  

不过最难的地方也在这里。

AI 只会生成还不够。它必须能验证，能理解团队上下文，能知道什么时候该打扰人、什么时候不该打扰人，能把自己学到的东西沉淀成可复用能力。

OpenAI 做 Codex 的方向，本质上是在回答一个问题：怎样让 AI 不只是聪明，而是默认有用？

如果这个问题被解决，最大的瓶颈就不再是我们每分钟能打多少字，也不是我们一天能给 AI 下多少命令。到那时，智能体才真正从“工具”变成“队友”。
