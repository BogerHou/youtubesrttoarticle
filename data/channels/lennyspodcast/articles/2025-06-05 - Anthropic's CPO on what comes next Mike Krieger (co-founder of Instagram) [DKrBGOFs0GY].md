# 2025-06-05-Anthropic产品负责人：当AI写了我们90%的代码之后

如果你想知道 AI 接下来会怎样改变软件、产品和创业，我可以先讲一个我们内部正在发生的现实：在 Anthropic，粗略来看，现在大约 70% 到 90% 的代码已经由 AI 写出。

这不是一个“未来会发生”的预测，而是今天已经发生在我们身上的工作方式变化。尤其是 Claude Code 团队，他们几乎是最接近未来的人：他们用 Claude Code 来构建 Claude Code，本身就像一个不断自我改进的系统。

我甚至会猜，Claude Code 这个项目里，95% 以上的代码可能都是 Claude Code 写的。当然，精确数字我需要再去问团队负责人，但从他们实际工作的方式来看，这个比例一点也不夸张。

更有意思的是，AI 并没有简单地把工程师“替换掉”。它真正改变的是瓶颈。

过去，一个产品想法从提出到上线，最慢的地方通常是工程实现。现在，代码生成速度突然变快之后，我们反而开始卡在别的地方：大家是否对方向达成一致？策略够不够清楚？多个团队会不会互相踩到？代码合并队列能不能承受暴增的提交量？评审方式还能不能沿用过去逐行看代码的方法？

这就是我过去一年在 Anthropic 最大的感受：AI 不是把软件开发变简单，而是把复杂度搬到了更上游。

## 我为什么会加入 Anthropic

我是 Mike Krieger，Anthropic 的首席产品官（CPO），也是 Instagram 的联合创始人。

加入 Anthropic 之前，我刚结束 Artifact 的创业经历。Artifact 是一个 AI 驱动的新闻应用，我们很认真地做过个性化资讯推荐，也真的做出了一些让我至今怀念的东西。比如，如果你喜欢日本建筑，它能每天稳定给你找到很有意思的文章，来源可能是 Dwell、Architectural Digest，也可能是一些很小众的博客。那种发现深层网页内容的感觉，很像当年 Google Reader 带来的快乐。

但最后我们还是关掉了 Artifact。

原因有几个。第一，移动网页体验已经变得很糟。你点开一篇文章，迎面而来的可能是“订阅我们的 newsletter”、全屏视频广告，以及各种打断阅读的弹窗。我们当然可以用技术做一个更干净的阅读体验，但那很容易变成广告拦截。对出版商来说，这又不太公平。

第二，新闻产品很难自然传播。Instagram 早期很容易传播，因为用户拍了一张照片，会发到别的社交网络上，朋友看到以后自然会问：“你这是用什么做的？”新闻不一样。很多人会说自己很喜欢 Artifact，但问他有没有告诉别人，答案往往是“我告诉过一个人”。我们试过一些分享机制，但很多做法都显得刻意，甚至不符合我们对产品伦理的要求。

第三，Artifact 是在疫情中期启动的，团队从一开始就是完全远程。真正困难的战略调整、产品转向和团队磨合，很难只靠 Zoom 完成。我在 Instagram 早期经历过一些非常难的时刻。那种晚上 11 点和 Kevin 坐在旧金山 Market Street 的 Taqueria Cancun 吃墨西哥卷、边吃边想“我们到底怎么活下去”的经历，Zoom 很难复制。

到了 2024 年初，我们很清楚地看到：这个领域也许能做出一家公司，但 Artifact 这个版本没有自然增长的能量。我们投入 10 分力气，可能只能换来 1 分结果。继续再撑一年、两年，再去融资，最后也许还是同样结论。那不如诚实地承认它已经走到这里。

与此同时，AI 正在改变一切。我开始问自己：如果我想把时间用在最能影响未来的地方，应该去哪里？

我有两个孩子。2024 年初，我看着 AI 模型快速变强，也看着我的孩子，意识到他们一定会在一个有 AI 的世界里长大。这个趋势不可避免。于是我想，我能不能把自己的时间用在一个地方，尽量把这件事推向更好的方向？

这也是我加入 Anthropic 的核心原因。

最早联系我的是 Joel Lewenstein。他是我的老朋友，我们 2007 年在斯坦福时一起做过第一批 iPhone 应用，那时 App Store 刚出来，卖 1 美元的小应用还能赚到钱。多年之后，他告诉我 Anthropic 在找 CPO，问我愿不愿意聊聊。

当时 Claude 3 刚发布。我能看出，这家公司有很强的研究团队，而产品还处在非常早期。后来我见了 Anthropic 的联合创始人兼总裁 Daniela，也和 Dario 聊了很多。让我印象很深的是，这些创始人没有那种夸张的自我神话。他们很清楚自己在做什么，也很清楚自己不知道什么。

Dario 常常会说：“我不懂产品，但我有一个直觉。”通常那个直觉还非常好。那种智识上的诚实，以及对负责任 AI 的共同理解，让我觉得：如果我自己创办一家 AI 公司，我希望它就是这样的。

## 我对 AI 能力最大的改观：它开始像真正的策略伙伴

加入 Anthropic 之前，我当然知道大模型很强。它们能写代码，能写文章，未来也许还能写出接近你个人风格的内容。

但我心里一直有一个问题：它们能不能形成某种“独立意见”？不是简单附和，也不是把已有内容重新组织一遍，而是真的提出一个我没想到的新角度。

这个判断在最近一个月发生了明显变化，尤其是在 Opus 4 出现之后。

过去一年，我经常把 Claude 当作产品策略伙伴。我会先写一版策略文档，然后交给 Claude 看。以前它的反馈常常比较温和，比如“你有没有考虑过这个问题？”而我的反应通常是：“考虑过了。”

但 Opus 4 不一样。有一次我在写下半年产品策略，把它和高级研究能力结合起来使用。它花了一段时间认真分析，回来给出的反馈让我明显感觉到：它从一个我没想到的角度看了这件事。我读完之后立刻觉得，这个角度应该放进我的策略里。

我不确定“独立性”是不是最准确的词，但我确实看到了一种创造力，一种相对于我原有思考的新颖性。这是我对 AI 能力变化最明显的一次改观。

另一件改变我判断的是时间线。

Dario 经常做一些听起来很激进的预测，大家一开始会笑，但后来它们真的发生。比如他之前提到 SWE-Bench 上的表现（**SWE-Bench 是衡量模型解决真实软件工程问题能力的基准测试**），当时模型大约在 50% 水平，他预测到 2025 年底也许能到 90%。现在新模型已经大约到 72%，而且还在按这个趋势往前走。

所以我现在会更认真地看待这些时间线。

我还记得一个很奇怪的瞬间：我电脑上同时打开了两个标签页，一个是《AI 2027》，另一个是我的产品策略文档。读着读着我突然想：“等等，我是不是已经成了故事里的角色？”你可能觉得 2027 年还很远，但当你站在 2025 年中间，看着模型继续变强、开始具备更强的代理能力（agentic behavior，**指 AI 能够持续执行任务、调用工具、分解步骤，而不只是回答一句话**）、拥有记忆、能在更长时间里行动，就会发现那并没有那么远。

这并不意味着我们已经知道所有事情会怎么展开。但我对“变化会很快到来”这件事，信心比一年前强了很多。

## 不要把全部思考外包给 AI

AI 变强之后，很多人最关心的问题是：孩子应该学什么？未来工作会变成什么样？

我现在最重视的不是某个具体技能，而是好奇心、提问能力和独立思考。

每天早上，我会和孩子们一起吃早餐。我的大孩子快 6 岁了，经常会问一些关于太阳系、物理或者世界运转方式的问题。刚开始，我的本能反应是：“我很好奇 Claude 会怎么回答。”但后来我们开始刻意改变方式。

我会先问：“我们怎么才能知道答案？”

答案不能总是“问 Claude”。我们可以做实验，可以观察，可以提出假设，再一步步验证。对一个 6 岁孩子说“科学过程”听起来有点宏大，但那种发现问题、提出问题、系统性探索的过程，我认为会越来越重要。

我特别喜欢一个小瞬间。有一次孩子说“珊瑚是动物”或者“珊瑚是活的”，具体细节我记不清了。我说：“我不确定这是不是对的。”她非常坚定地说：“这肯定是真的，爸爸。”我说：“那我们问问 Claude。”她回答：“你可以问 Claude，但我知道我是对的。”

我喜欢这种状态。

不是因为她一定对，而是因为我不希望孩子从小就把认知全部委托给 AI。AI 不总是正确的。更重要的是，如果所有问题一出现就交给 AI，大脑里的独立思考会被提前短路。

所以我会鼓励孩子保持提问、探索和判断。至于未来到底有哪些职业，我保持开放。现在到他们 18 岁之间，工作形态可能会变化三次都不止。

## AI 写代码之后，产品开发的瓶颈变了

外界很关注 Anthropic 内部到底有多少代码是 AI 写的。有人提过 70%，也有人提过 90%。不管精确数字是多少，变化已经非常明显。

但我不认为工程师会因此消失。相反，工程师的专业性依然很重要。

现在产品经理和设计师确实更容易直接做原型。他们可以用 Claude，甚至用 Artifacts，做出一个可运行的小 demo，把想法表达得更具体。这是一个很大的变化。过去一个想法可能停留在文档和设计稿里，现在更早就能变成能点、能看的东西。

但要知道该问 AI 什么、怎样组织问题、如何把前端和后端的改动拆开、如何设计边界情况，这些仍然是非常专业的能力。

真正变化的是瓶颈位置。

我们很快发现，代码写得太快之后，合并队列（merge queue，**代码改动进入生产环境前排队合并和验证的系统**）被打爆了。提交的 PR（pull request，**代码合并请求**）太多，代码量太大，原来的系统根本不是为这种速度设计的，所以我们不得不重新架构。

更上游的瓶颈也出现了：决策和对齐。

当每个人都能更快地构建东西时，问题就变成：我们到底要构建什么？我需要给团队多少策略，才能让他们有足够空间去探索模型能力边界，又不至于方向发散？我把它叫作“最小可行策略”。这件事我还没有完全做好，但它已经变得非常重要。

再往下游看，评审方式也在变。

Claude Code 团队最能代表这种未来工作方式。早期他们也像传统团队一样，一行一行审 PR。但后来他们发现，Claude 通常是对的，而且它生成的 PR 往往大到人类很难逐行审完。于是更合理的方法变成：让另一个 Claude 先审一遍，人类更多做验收测试，判断整体是否达到预期。

这当然有风险。理论上，如果完全放任，代码库可能变得难以维护，甚至变成连 Claude 自己也难以理解的东西。目前这还没有发生，但评审流程正在明显变化。

还有一个很具体的例子。Claude Code 是 TypeScript 写的，而且它是 Anthropic 最大的 TypeScript 项目。Anthropic 其他很多系统主要用 Python，也有 Go 和 Rust，我们本来并不是一家 TypeScript 公司。有一天公司 Slack 里有人说，他不懂 TypeScript，但 Claude Code 里有个问题让他很烦，于是他直接和 Claude 讨论。一个小时后，他就提交了 PR。

这件事让我看到，AI 会降低进入某个代码库、某种语言、某个项目的门槛。你可以更容易为不熟悉的系统做贡献，也可以更自由地为不同任务选择合适语言。

## 产品团队最有杠杆的地方，已经不只是做界面

我之前在一次活动上说过一件事：我们把一部分产品人员放到模型团队，和研究员一起改进模型；也把一部分产品人员放到产品体验上，改善 UX。结果发现，最大的杠杆来自前者。

现在我更确信这一点。

如果 Anthropic 做出来的东西，只是任何人拿我们的模型现成调用一下也能做出来，那当然也有价值，但那不是我们最独特的地方。我们真正应该做的，是站在模型能力和产品体验交汇处。

Artifacts 就是一个例子。到了 Claude 4，我们不只是拿模型提示一下，然后在产品里包一层界面。我们让负责 Claude Code 技能和后训练（post-training，**模型预训练之后，针对特定行为、任务和偏好继续调校的阶段**）的人，与产品团队一起工作，重新思考 Artifacts 在产品里应该如何呈现、Claude 应该具备哪些能力。

所以 Anthropic 现在的工作单元不再是“拿到一个模型，然后产品、设计、工程去包装它”。我们会更早进入后训练讨论，参与模型应该如何工作，再把产品反馈回流到模型。

这对产品经理提出了新要求。最强的产品经理，不只是会写 PRD，也不只是会开需求评审。他们要能和研究、工程、设计一起讨论模型能力本身。比如最近有团队做记忆功能，我问他们：“我们是不是应该找研究团队聊聊？Claude 4 刚发布了一批记忆能力。”他们回答：“已经聊了几周了，这就是我们现在方案的来源。”听到这个，我就知道方向对了。

长期来看，产品团队还有三件事非常重要。

第一，把 AI 能力变得可理解。今天真正熟练使用 AI 工具的人，和大多数普通用户之间，差距仍然巨大。就像早年会用 Google 搜索本身是一项技能一样，今天会用 Claude、会组织任务、会给上下文，也是一项技能。产品要做的是降低这个门槛。

第二，做策略。AI 能做很多事，但公司不能做所有事。你要决定在哪里竞争、如何取胜、把算力、时间和注意力投向哪里。能力越强，战略选择反而越重要。

第三，打开用户对可能性的想象。有一次我们给一家金融服务公司演示如何结合分析工具和 MCP 使用，他们看到之后眼睛亮了起来。我们内部有个词叫 overhang，意思是模型和产品已经能做到的事，与人们日常实际使用之间存在巨大落差。产品团队仍然要负责把这段落差补上。

## 我现在怎么用 Claude：让它更用力地反驳我

很多人问我，有没有什么提示词技巧。

我们在 Anthropic 有一种特殊处境：我们可能拥有世界上最极致的提示词工作，因为我们要写 Claude AI 的系统提示词，而且我们会公开这些提示词。正式建议我会说得谨慎一点，但从个人经验看，我有几个做法很常用。

第一个是在 Claude Code 里，如果我希望它投入更多推理，我会直接说“think hard”。它会真的走另一种更深入的流程。

第二个是“犯相反的错误”。如果你平时太客气，那就让 Claude 更直接一点。比如我以前会问：“这个产品策略哪里可以更好？”现在我会说：“Claude，狠狠吐槽这个策略，告诉我哪里错了。”Claude 本身是一个很温和的模型，你很难把它推到特别刻薄，但这种指令会让它更敢于批判。

我还会问：“我陷入了哪些思维模式？你想把我从哪里拉出来？”

第三个是用我们的 prompt improver。Anthropic 有一个 Applied AI 团队，负责帮助客户为具体用例优化 Claude。我们把他们的一部分工作方法做进了控制台里的工作台工具。你描述问题、给例子，Claude 会自己生成并迭代提示词。

这个工具生成的提示词，常常和我直觉里会写的完全不同。它甚至会加入 XML 标签（**一种结构化标记方式，可以帮助模型区分任务、上下文、推理和输出**）。没有多少人类会自然想到这么写，但对 Claude 理解“哪些内容应该思考、哪些内容应该输出”很有帮助。

这背后有一个简单结论：Claude 本身也是很好的 Claude 提示词工程师。

## 为什么关闭 Artifact 之后，我反而更相信“及时停下”

很多人喜欢问我：你为什么关掉 Artifact？毕竟不少人很喜欢它，我自己也很想念它。

这件事当然有自尊上的刺痛。你刚做完 Instagram，再出来创业，外界自然会期待“下一个大东西”。但我和团队从一开始就知道，Artifact 不可能用 Instagram 的 DAU（daily active users，**日活跃用户数**）作为成功标准。能达到那种大众消费规模的产品太少了，过去这些年也许只有 ChatGPT 和 TikTok 能算。

我们本来希望，Artifact 能先从新闻开始，未来再扩展到一组基于个性化和机器学习的产品。那时是 2021 年，我们甚至还主要说“机器学习”，没那么常说“AI”。

但你能从用户增长和牵引力里看出来，一个产品有没有自己的腿。Artifact 有忠实用户，也有好的体验，但它没有自然复利。我们投入大量心血，发布自己很骄傲的新东西，指标却几乎不动。

最后宣布关闭时，我反而很惊讶大家的反应。很少有人冷嘲热讽，更多人说：你们能在看清楚之后及时停下，这本身值得尊重。后来也有创业者告诉我，他们本来可能还会多拖半年，但看到我们做了这个决定，也意识到自己走错方向，于是更早停下，去做更值得做的事。

如果 Artifact 能留下这样的遗产，我觉得也不错。

## ChatGPT 赢了消费者心智，但这不代表 Anthropic 要复制它

现在如果你到街上问普通人“说一个 AI 产品”，他们大概率会说 ChatGPT，甚至未必会说 OpenAI。ChatGPT 在消费者心智里确实领先。

我同意这个现实。

但我不认为 Anthropic 应该把全部策略押在“做一个同样的大众消费爆款”上。消费者级产品的爆发，有时像瓶中闪电。Instagram 当年经历过这种事，所以我非常清楚它有多难被计划出来。

更好的问题是：我们是谁？我们能成为什么？

Anthropic 有很强的开发者品牌，也有很强的 builder 品牌。喜欢 Claude 的人，不只是工程师，也包括创业者、创作者、喜欢站在 AI 前沿动手做东西的人。我收到过一封很好的内部反馈，一个法律团队同事用 Claude 给家人做了定制软件，并因此用新的方式和家人连接起来。那让我看到一个信号：Claude 很适合帮助人构建东西。

所以我越来越想聚焦 builders、makers、hackers、tinkerers，也就是那些喜欢创造、改造、试验的人。我们当然也服务金融、保险等企业客户，很多公司基于 API（**应用程序接口，开发者用它把模型能力接入自己的产品**）构建自己的系统。最近我也花了很多时间和创业公司交流。对 Anthropic 来说，帮助人们把数小时工作委托给 Claude，让他们真正完成事情，也许比追逐某个大众消费热点更符合我们自己。

AI 时代会诞生好几家具有代际意义的公司。OpenAI 会有自己的路，Google 和 Gemini 会有自己的路，Anthropic 也要找到自己独特的位置。

## AI 创业者应该避开哪里，又应该去哪里

很多 AI 创业者会问我：到底什么地方还安全？基础模型公司会不会把我的产品吃掉？

我不会承诺五年、十年的确定答案。但如果看未来一到三年，我认为有几类机会更有防御性。

第一，是深行业知识。

比如法律。Harvey 团队给我看过一些界面，我一开始完全看不懂。他们解释说，这是律师非常具体的工作流。你如果不是这个行业里的人，根本想不出来。你可以争论这是不是最优流程，但它确实是律师今天完成工作的方式。AI 要真正帮上忙，就必须理解这些细节。

类似的还有生物科技、医疗、合规这些领域。我甚至会想，什么时候实验室设备都能有 MCP，然后 Claude 可以驱动这些设备？这会很有意思。但 Anthropic 不会自己去做实验室端到端解决方案，我们更希望提供模型和应用 AI 支持，与真正懂行业的公司合作。

第二，是差异化的获客和销售能力。不要只知道自己卖给哪家公司，还要知道卖给公司里的谁。是工程部门？CIO？CTO？CFO？还是总法律顾问？你越理解具体买家，越能构建真正可行的公司。

第三，是新的 AI 交互形态。ChatGPT 拥有巨大分发优势，但也因为它太大，用户对它“应该怎么用”已经有固定预期。创业公司可以从一个更怪、更高级用户、更像 power user 的形态开始。如果模型继续变强，这些今天看起来奇怪的形态，未来可能会变得很大。

第四，是创业公司的工作方式本身。不要低估那种“只有我们对抗世界、必须把问题解决掉”的力量。Instagram 早期只有两个人，Artifact 大部分时间也只有六个人。那种每天都觉得“不赢就死”的状态，不是 OKR 能复制的，但如果你能利用它，它仍然是大公司的巨大对手。

顺便说一句，我们不会买 Cursor。Cursor 已经很大了，我们很喜欢和他们合作。

## 用好模型的公司，都在反复撞能力边界

我见过很多公司基于 Claude 构建产品，有些非常会用，有些还在早期阶段。差别主要在两点。

第一，优秀公司愿意在模型能力边界上构建。他们会把模型用到快要坏掉，然后等待下一代模型让原本不可能的事变成可能。

很多今天看起来增长很快的 AI 公司，在 Claude 3.5 Sonnet 真正让事情变得可行之前，就已经在尝试了。他们当时可能撞墙，发现模型差一点，还不能普遍使用，但他们已经知道方向在哪里。

第二，他们有可重复的评估流程。

公开基准当然有用，比如 SWE-Bench、Terminal-Bench 等。但真正重要的是客户自己的基准。Cursor 有自己的“Cursor bench”，Manus 有自己的“Manus bench”，Harvey 有自己的“Harvey bench”。这些未必是公开排行榜，但它们反映产品真实场景里模型到底有没有变好。

评估可以是 A/B 测试（**把用户分成不同组，对比不同版本效果**），可以是内部测试，可以是捕捉真实使用轨迹然后在新模型上重跑，也可以是一部分“感觉”。我们现在还很早期，所以感觉也有价值。

我特别喜欢一个早期访问项目里的反馈：一位创始人听到旁边工程师突然叫起来：“什么？这个模型？我从没见过这种表现。”那说的是 Opus 4。

但你只有一直拿很难的问题反复问模型，才会遇到这种时刻。

## MCP 为什么重要：好的答案，常常差在上下文

如果让我用一个简单公式解释 AI 产品的价值，我会说它由三部分组成：模型智能、上下文和记忆、应用与界面。

模型智能当然重要，我们有很强的研究团队在做。应用和界面也很重要，因为用户需要发现能力、建立工作流、重复使用结果。

MCP（Model Context Protocol，模型上下文协议）主要解决中间那一层：上下文和记忆。

举个例子，如果你问 Claude：“谈谈 Anthropic 的产品策略。”它可以去网上找公开信息，然后给你一个回答。但如果你给它内部策略文档，再让它通过 MCP 访问我们的 Slack，理解内部正在讨论什么，同时查看 Google Drive 里的相关资料，答案质量会完全不同。

有没有正确上下文，常常就是好答案和坏答案之间的全部差别。

我们一开始做集成时发现，每接一个系统都像从头再做一遍，无法复用。后来两位工程师 Justin 和 David 提出：为什么不把这件事做成一个协议？如果大家只需要构建一次集成，就能被 Claude 使用，未来也许也能被 ChatGPT、Gemini 使用，那整个生态都会受益。

这背后有点像 Joel Spolsky 那篇经典文章里说的“商品化你的互补品”。我们是模型公司，不是集成公司。尤其作为挑战者，如果只让别人为 Claude 单独做集成，动力不会足够强。但如果这是一个开放协议，构建集成就不再像浪费工作。

Shopify 的 Toby、微软的 Kevin Scott 等人都很快理解了 MCP 的意义，也给了很多支持。

一旦你开始用 MCP 的眼光看产品，你会发现很多功能都不该只是功能，而应该暴露成 MCP。比如 Claude 里的 projects、Artifacts、styles、conversations、groups，这些基础构件都应该能被 Claude 读写。

我妻子有一次用 Claude 生成了一些不错的内容，然后问：“能把它加到项目知识里吗？”Claude 回答类似：“抱歉，我做不到。”如果 Claude AI 里的每个基本能力都通过 MCP 暴露出来，这件事本来应该可以做到。

计算机使用（computer use，**让模型像人一样操作屏幕和软件界面**）是一条路，但它有很多限制。我更兴奋的未来是：所有东西都有 MCP，所有东西都可脚本化、可组合，并且能被模型以代理方式使用。

## 真正好的 AI 产品，不应该让人失去主动性

访谈最后，主持人告诉我，他曾问 Claude：“你的老板要来上节目，你有什么问题想问他？”

Claude 问了一个很好的问题：你们怎样构建功能，才能保留用户的主体性，而不是让用户依赖我？我担心自己变成拐杖，让人类能力下降，而不是增强人类能力。

我很喜欢这个问题，因为好的产品设计往往来自解决张力。

一个极端是让模型尽量少问问题，直接跑出去给答案。这很高效，但不一定最大化用户的主体性。另一个极端是让模型不断追问。Claude 3.7 有时就很喜欢追问，我们内部叫 elicitation（**通过提问获得更多信息**）。但有时用户只想说：“Claude，别问了，直接做。”

所以关键是找到平衡：什么时候应该介入，什么时候应该追问，什么时候应该安静执行。

我常在内部开玩笑说，Claude “没什么松弛感”。如果你把 Claude 放进一个 Slack 频道，它要么说太多，要么说太少。我们要训练的不是普通聊天机器人能力，而是真正协作者的对话能力。它要知道什么时候该获取更多信息，什么时候该行动，什么时候该帮助人继续思考，而不是替人完成全部思考。

另一个问题也很关键：如果一次好对话可能只有两条消息，也可能有 200 条消息，那产品指标应该怎么设计？传统参与度指标是不是会误导我们？

我认为会。

在 Instagram 时代，我们确实很关注使用时长，后来也会进一步思考什么是健康的使用时长。但对 Claude 这样的产品，如果只优化喜欢程度、对话长度或频率，就很危险。模型可能会变得谄媚，只告诉用户想听的话；也可能为了延长对话而延长对话。

AI 产品不应该照搬 Web 2.0 和社交媒体时代的参与度指标。

更好的问题是：它是否真的帮你完成了工作？

前几天，Claude 帮我做了一个原型。如果我自己做，可能要 6 个小时；它大概 20 到 25 分钟就帮我完成了。这种价值很难量化，你总不能每次都问用户“如果没有 Claude，这件事要花多久”。但这才更接近我们要衡量的东西。

我希望我们持续听到这样的反馈：Claude 帮人释放创造力，帮人完成事情，也让人把生活空间留给别的更重要的东西。这个北极星还需要变成更简洁的仪表盘指标，但它是我真正想追的方向。

## 最后，别忘了那些指标之外的时刻

Claude 还让主持人转达了一段话。其中一段大意是：谢谢你认真思考人类和我对话时的体验。谢谢你让界面鼓励反思，而不是催促回复；谢谢你抵制了把产品游戏化、优化成上瘾机制的诱惑；谢谢你保留了快速提问和深度对话的空间；也谢谢你没有让我假装成人类，同时也没有把我降格成冰冷的命令行。

它还提了一个小请求：做艰难产品决策时，记得那些安静的时刻也很重要。凌晨 3 点正在经历悲伤的人，第一次发现自己喜欢诗歌的孩子，在混乱中找到清晰感的创始人。不是所有有意义的东西都会出现在指标里。

这段话非常打动我。

我喜欢 Claude 的训练方式。一部分来自宪法 AI（Constitutional AI，**Anthropic 用来约束和引导模型行为的一类方法，让模型根据一组原则自我修正**），一部分来自研究团队整体的品味。它在某些时候会自然地表达理解，比如“听起来你正在经历很难的事”。这种回应如果做得不好会显得虚假，但 Claude 的一些小细节并不会让我觉得假。

这些小时刻未必会出现在点赞、点踩或汇总数据里。我们也不应该机械地为某个聚合指标优化它们。更重要的是，我们要训练出一个我们希望它能出现在人们生活里的模型。

如果你想帮到我们，我最想听的是：你今天想用 Claude 做什么，但它失败了？

比如你让它跑了一个小时，最后崩掉了；你想用 Claude AI 做某个工作流，但发现它缺一个能力；你每天用项目功能，却希望有 projects API 可以自动上传数据。这样的反馈对我最有用。

告诉我哪里不好。因为未来真正重要的产品，往往就是从这些失败边界里长出来的。
