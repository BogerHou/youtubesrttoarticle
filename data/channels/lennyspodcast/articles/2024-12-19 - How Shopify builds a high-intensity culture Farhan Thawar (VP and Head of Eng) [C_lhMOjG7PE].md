# 2024-12-19-Shopify如何打造高强度文化

如果你只看 Shopify 的规模，很容易把它想成一家已经进入稳定期的大公司：成立将近二十年，员工超过一万人，远程办公，业务横跨线上商店、支付、物流、线下销售点系统和开发者生态。

但我在 Shopify 负责工程团队时，看到的不是一家“慢慢变稳”的公司。恰恰相反，我们一直在问一个问题：一个万人级组织，怎样还能保持创业公司式的速度、紧张感和判断力？

我自己的答案，后来可以归成三件事。

第一，遇到选择时，尽量选更难、但更能学习的路。

第二，强度不是让大家工作更久，而是在同样的时间里创造更高密度的产出。

第三，招聘不能只靠面试，要尽量接近真实工作，用真实产出来判断一个人。

这三件事听起来都不复杂。但真正做起来，很多做法都反直觉。

## 选难路，不是为了吃苦，而是为了不白输

我经常给求职者一个建议：如果你有几个选择，优先选那条更难、但更有学习密度的路。

原因很简单。

如果你选了容易的事，成功了当然不错；但如果失败了，你通常什么也没学到。你只是走了一条轻松的路，然后发现它没通。

如果你选了难的事，就算最后没成功，你也已经赢了一部分。因为你做过困难的工作，和更聪明的人一起解决过问题，积累了真正可迁移的能力。

我见过很多找工作的人。他们说自己每天投十份简历。我会问：这件事真的让你学到了什么吗？

如果你想去 Shopify，为什么不打开 Shopify 的 API 文档，自己做一个小应用？做一个 Shopify app，做一个后台扩展，或者围绕 Shopify 平台搭一个工具。

就算你最后没有拿到 Shopify 的工作，你也多了一个 GitHub 项目，多了对产品和平台的理解，也多了一个可以展示的真实作品。这些东西可能帮你去下一家公司。

这就是难路的价值。

我在读书时也这样做。读计算机科学时，我又加了电气工程辅修。读 MBA 时，我又选了金融工程辅修。它们更难，成绩可能更差，但聪明人往往在那里。更重要的是，我会被迫进入一个自己不舒服的房间，然后承认：“我不懂。”

很多人把“看起来聪明”当成目标。我反而觉得，能反复在公开场合看起来很笨，是一种超能力。

我问过很多很蠢的问题。有人当面说过：“这是我听过最蠢的问题。”最近还有人跟我说：“这个问题我已经给你解释三遍了。”

我的反应通常是：好，那我还是没懂，我们继续。

我不是为了惹人烦，而是为了弄明白。很多时候，我问出所谓“蠢问题”后，旁边的人会说：“其实我也想问。”

这个习惯可能来自我年轻时做零售。你靠提成赚钱，就必须主动走向顾客。有人不买，你不能让一天都毁掉，只能转向下一个人。被拒绝多了，你就知道，“丢脸”本身没有那么可怕。

## 找到能互补的人，而不是逼自己什么都擅长

回头看我的职业生涯，我发现一个有趣的模式：我每个十年都和不同的亿万富翁共事过。二十多岁时是 Joe Lemond，三十多岁时是 Chamath，这个十年是 Shopify 的 Tobi。

我当然不是按这个目标规划人生的。但这些人有一个共同点：他们对未来有一种“不合理”的长期判断。

他们会看十年、十五年、二十年以后世界会怎样。我自己并不擅长凭空看见那种远景。但我擅长每周把事情往那个方向推进 1%。

这让我意识到：我不需要强迫自己成为所有方面都最强的人。愿景不是我的核心优势，执行和推进才是。所以我应该去找那些“不合理”的人，然后和他们“API 兼容”。

2015 年我准备创业时，我甚至写过一张清单：多伦多有哪些我认识的“不合理”的创业者？我一个个去见，判断我们能不能合作。最后，我选了其中一个人一起开公司。

这也影响了我选择工作的方式。

很多东西都会干扰判断：头衔、公司名、薪酬、外界眼光。如果你没有提前写下自己的框架，很容易被这些东西带走。

我有一套自己的工作选择框架。每当机会出现，我会问它是否符合这个框架。甚至有一次，一个招聘人员反复联系我，我直接把我的框架发给他，说：这个机会不符合。

这套框架也让我离开过不合适的岗位。

在 Xtreme Labs 被 Pivotal 收购后，我的角色从管理全球最大的结对编程办公室，慢慢变成了 field CTO（面向客户和市场的技术负责人）。这段经历有学习价值，但它已经不符合我最在意的东西：我不再和最聪明的人一起解决最重要的问题，也没有足够大的影响力。

于是我辞职了。没有先找好下一份工作。

这听起来很突然，但对我来说很清楚：一旦框架被违反，就该重新选择。

## 强度不是加班，而是每分钟做更多

很多人谈高强度文化，第一反应是“多加班”。我不这么看。

一小时就是一小时。你的一小时和我的一小时，流逝方式一样。关键是，在这同一小时里，我们消耗了多少能量，创造了多少真正有价值的东西。

我更关心的是每小时释放多少“千焦能量”，而不是一天工作多少小时。

我以前待过一种公司：大家看起来工作十二小时，但中间打桌上足球、喝咖啡、闲聊。时间被拉长了，但密度并不高。

我更喜欢把工作压缩进更清醒、更专注的八小时里。这样你仍然可以下班后带孩子去踢球、过自己的生活，但工作时间里，真正把事情往前推。

这就是我为什么喜欢结对编程（pair programming）。

结对编程不是“两个人各写一半代码”。它是两个人共用一台机器，两个键盘、两个麦克风、两个屏幕，但一个上下文。远程时，我们会用 Tuple 这样的工具。

外人最常见的疑问是：两个工程师用一台电脑，不是产出减半吗？

我的回答是：如果你用“敲了多少行代码”衡量产出，那确实误解了工程工作。

真正限制产出的，不是手指敲键盘的速度，而是能不能找到优雅的解决方案。代码不是资产，很多时候代码是负债。它会长期存在、需要维护、需要别人理解。

Tobi 早年和 Shopify 的 CTO Cody 结对编程时，有一个极端做法：设一个一小时计时器。如果一小时内没完成，就删掉所有代码，只保留测试，然后重写。

如果刚好超时一分钟，也删。

背后的判断是：如果我们无法在一小时里把这个功能的代码写清楚，说明设计可能不对。正确的设计，应该能更清晰、更短、更容易解释。

在 Pivotal Labs，也有一种更强硬的文化：如果你的搭档某天生病，你自己写了一堆代码，第二天搭档回来后，可能会先删掉这些代码，然后两个人重写。

这不是浪费。刚写完时，恰恰是你最懂问题域的时候。此时重写，往往能得到更简洁的方案。

当然，删除自己刚写的代码非常难。人类有沉没成本偏差（**已经投入的时间和精力，会让我们不愿意放弃，即使放弃才是更优解**）。但如果你能克服这一点，代码质量会明显提高。

结对编程还有一个被低估的好处：它让人无法多任务。你不会一边写代码一边刷 Slack、看社交媒体，因为另一个人就在同一个问题里。两个人一起思考、讨论、删除、重来，强度会非常高。

刚开始结对编程的人，前几天晚上可能会睡十到十二个小时。因为它真的累。

但这种累不是无意义的消耗，而是高密度的专注。

## AI 让每个人都多了一个搭档，但人和人仍然重要

今天，GitHub Copilot、Cursor 这样的 AI 工具已经在改变编程方式。某种意义上，AI 已经成了你的结对程序员。

你可以对着 Cursor 说话，用 Whisper 把语音转成指令：“帮我写一个 React 组件，它要做这些事。”AI 生成后，你再说：“不对，我的意思是这样。”整个过程像和一个不会累的搭档来回推演。

这非常好。它让人不再必须独自写代码。你想尝试一门新语言，AI 能帮你补上 API 和语法细节。

但我仍然认为，最强的组合不是“一个人加 AI”，而是“两个人加 AI”。

AI 可以快速生成代码。两个人可以一起判断：这段建议是不是对的？有没有启发？应该接受、修改，还是直接删掉，自己重写？

未来几乎每个工程师都会以某种形式结对编程。区别只是你的搭档是人、AI，还是两者都有。

## Shopify 的节奏：每周更新，每六周复盘

Shopify 的强度不是靠喊口号维持的，而是靠节奏。

我们有一个内部工具叫 GSD，也就是 get shit done。每个项目每周都会向全公司更新进展。更新里可能有文字，也可能有产品视频，说明上周以来发生了什么、学到了什么、下一步是什么。

这其实是在组织层面利用帕金森定律（**工作会自动填满你给它的时间**）。如果你每六个月才检查一次，很多事情就会拖成六个月。如果每周都要展示进展，团队自然会问：这一周我们到底推进了什么？

我们还有 OK1 和 OK2。

OK1 通常在 director 层级，关注方向是否对齐。OK2 通常在 VP 层级，关注这件事是否符合更大的架构、业务背景和行业变化。

更重要的是，每六周我们会做一次 review。团队会和直接领导层一起过路线图、资源分配、项目状态，也会和 Tobi 一起看。

这是一笔很大的时间投入。大家会集中几天，把公司里的项目一个个过掉：这个项目在做什么？资源够不够？为什么这么慢？是不是想太复杂？是不是被某个基础设施卡住了？

六周这个周期很有意思。它短到大家还记得上下文，长到一个十几人工程团队足够做出明显进展。

更关键的是，反馈不会等到下一个六周。

有一次，我们在做一个和大语言模型（LLM）相关的 AI 项目，需要非常大的输出上下文窗口（**模型一次回答中能生成的内容长度**）。当时很多模型输入上下文很大，但输出上下文较小，团队以为必须把数据切块、缓存、再拼接。

我就在和 OpenAI、Gemini 等模型团队的共享 Slack 频道里问了一句：能不能给 Shopify 开更大的输出上下文？

一小时内，几个主流模型的限制就被提高了。项目立刻解堵。

这就是强度的真实来源。不是等流程慢慢走完，而是快速暴露问题、快速找到能解锁的人、快速继续前进。

## “微管理”不一定是脏词，关键是一起解决问题

很多人讨厌 micromanagement（微管理）。在 Shopify，这个词不一定是脏词。

当然，我们不想要那种 Dilbert 漫画里的老板：每天问“你做完了吗”，只是为了抓人偷懒。

但如果微管理的意思是：“我想和你一起解决这个问题，所以我需要经常看到你走到哪一步，然后把我知道的上下文和你知道的上下文合在一起”，那它就不是坏事。

我刚到 Shopify 时，Tobi 会和我做一对一。我当时想：你不用浪费时间，你雇我就是为了让我把问题拿走。

他纠正我说：你误解了。我们坐在一起，是为了共同解决问题。

这句话改变了我对管理的理解。

领导不是只做检查点。领导应该在关键问题上和团队“结对”。工程师掌握现场细节，领导掌握更大的上下文。两者合起来，才更容易做出正确判断。

## 我们每年随机删除一次会议

Shopify 还有一个很有名的做法：Meetingageddon，可以理解为“会议末日”。

每年某个随机时间，我们会删除所有超过两个人、只包含内部人员的周期性会议。也就是说，一对一不会删，面试和外部合作伙伴会议不会删，但内部的 recurring meeting 会被清空。

然后有两周冻结期，不允许重新添加周期性会议。你可以开一次性会议，但不能马上把它变回每周固定。

为什么这么做？

因为周期性会议有巨大惯性。它一直在日历上，大家就默认它应该存在。实际上，很多会议只需要开一次，或者应该换个频率，或者根本不需要。

删除会议后，我们看到了明显变化。

我会追踪工程团队里的 individual contributor（个人贡献者）每周花在会议里的时间。做完会议清理，再把大量公告从 Slack 迁到 Workplace 后，个人贡献者每周会议时间大约降到三小时。管理者大约是六到七小时。此前个人贡献者大约是五到六小时，管理者大约是十小时，整体下降了 50% 到 60%。

这对创作者非常重要。工程师、设计师、产品经理、机器学习工程师，都需要成块的深度工作时间。我们内部有一句话：Shopify 应该是 crafters' paradise，也就是“手艺人的天堂”。这里应该是人们练习手艺、提升手艺的地方。

Slack 很好，我们仍然使用它。但 Slack 是即时通讯工具，很容易让所有信息都变成打断。所以我们把公告、大范围状态更新，放到 Workplace 这种信息流里。你可以主动去消费公司信息，而不是被每条消息推着走。

Workplace 后来要停用，我们也还需要找下一个工具。但原则不变：即时沟通和组织信息流，不应该混在一起。

## 速度来自基础设施，而不是只做眼前功能

如果你想让组织长期更快，不能只盯着眼前功能。你要问：能不能构建一层基础设施，让以后很多功能都变得容易？

2020、2021 年加密货币很热时，很多 Shopify 商家想做 NFT gating。意思是，用户只有持有某个 NFT，才能进入店铺看到商品、价格并购买。

如果只是做这个具体功能，可能两三周能完成。

但 Tobi 问我们：如果要做一层平台，让任何人都能在一小时内做出 NFT gating，需要多久？

答案可能是两三个月。

他让我们做后者。

因为 NFT gating 只是一个用例。如果底层 API 和平台能力搭好了，别人会在上面做出我们没想到的东西。Tobi 把这叫“给油箱加油”。你把基础设施建好，以后很多人都能靠这箱油往前开。

他还有一种写代码方式很能说明问题：他会先写调用一个并不存在的 API 的客户端代码。写着写着，他会意识到：“这里应该有这样一个 API。”他先把理想的使用方式写出来，再回头实现服务端。

这也是 Shopify 做产品和工程的思路：不要只问“这个功能怎么做”，而要问“什么必须成立，才能让这个功能一天内就能做出来？”

因此我们会把工作分成三类：实验、功能、基础设施。

实验是为了学习。功能是建立在已有基础设施上的具体能力。基础设施则是让未来更多功能变得容易的底层能力。

很多团队会在第一个可行方案前停下。Tobi 经常提醒我们：任何问题都有无限多错误答案，也可能有一万个正确答案。不要在第一个正确答案前停下，要继续生成更多选项，找到更优的那个。

## 删除代码，是工程组织最被低估的加速器

我曾经发过一条消息：GitHub Copilot 已经为 Shopify 写了超过一百万行代码。很多人很兴奋。

但我真正想看到的是：GitHub Copilot 什么时候能帮 Shopify 删除一百万行代码？

能写代码不稀奇。真正像工程师的 AI，应该也能识别哪些代码不该存在。

Shopify 今年删除过数百万行代码。我们有 Delete Code Club，也会在一年两到三次的 hack days 里安排团队专门删除代码。甚至有手册告诉大家：怎样找到可以删除的东西。

听起来夸张的是，我们几乎总能找到一百万行以上可以删的代码。

这到底是好事还是坏事，可以讨论。但我更关注的是：团队愿意清理历史包袱。

代码库越简单，可靠性、性能、可维护性、可理解性都会变好。越多上下文能装进人的脑子里，开发就越快。

所以当有人说 Shopify 是全球最大的 Ruby 代码库之一时，我的反应不是骄傲，而是：我不希望我们出现在这个榜单上。我们不该争第一，我们应该争取掉到第一百名。

代码量不是目标。优雅才是目标。

## 用高保真演示减少误解

除了节奏、会议和代码，我们还非常重视 demo culture，也就是演示文化。

GSD 更新里，我们鼓励团队提供高保真更新。不只是截图，而是真正的演示。

截图看不出页面慢不慢、流程卡不卡、交互顺不顺。所以我们会用 Spin 这样的内部云开发环境，让别人点链接就能试。也会用 beta flag（测试开关）让别人直接在自己的店铺里打开新体验。

这样反馈会非常真实。

产品经理也会做 friction log（摩擦日志）：录屏走一遍流程，一边操作一边说哪里不顺、哪里慢、哪里让人困惑。

这比状态报告有价值得多。我们讨论的不是“项目是否按计划推进”，而是“这个体验到底怎么样”。

## 远程优先，但不等于永远不见面

Shopify 是远程优先公司，但我更愿意说我们是 90% 或 95% 远程。

因为我们会刻意安排线下体验。

比如 Shopify Summit，会把全公司聚在一起，有演讲、hack days、聚会、音乐和各种面对面活动。它的作用不是装饰，而是给“信任电池”充电。

信任电池（trust battery）是一个很有用的比喻：你和一个人的信任关系像电池，远程久了可能会消耗，见面合作可以充电。

我们还有 bursts。团队如果要原型验证、集中攻关，可以临时组织几个人去渥太华、多伦多、蒙特利尔或其他地方待几天，一起解决一个问题。

办公室也存在，但不是强制坐班。你想来就来。我以前一周去一次；Tobi 搬到多伦多后，我可能一周去三天。但这不是“每周五必须到办公室”的混合办公。

我们保持远程的核心原因很简单：我们想雇到世界上最好的人，而最好的人不一定住在办公室附近。

但远程不代表忽视线下关系。黑五网一（Black Friday Cyber Monday）时，我会鼓励工程领导层来多伦多，一起看图表、盯系统。hack days 时，我也鼓励大家去几个主要办公室聚在一起。

远程是默认，线下是有意设计的加速器。

## 组织里应该有一点火

Shopify 领导团队里有很多创始人型的人。他们通常都不耐烦、强度高、希望事情更快发生。

这种气质会影响公司。

我甚至认为，如果你没有两三个大项目处在“着火”的状态，你可能没有推得足够远。

这里的“着火”不是故意制造混乱，而是你在尝试超出舒适区的东西：新产品、新技术、新负责人、新节奏。只要你真的在拉伸组织，总会有边缘地带出现不确定性。

很多人以为舒适带来稳定。我的看法相反：不舒适反而带来稳定。因为你一直在学习，组织会更有韧性。

## 面试不是预测工作表现的好工具

聊招聘时，我最想先承认一个事实：面试并不是预测工作表现的好工具。

大家都知道这一点。每家公司都见过“面试很好，入职后不行”的人，也见过“面试一般，进来后非常强”的人。

我之前创业时招机器学习工程师，有两个候选人给我很深印象。

一个是博士，在大学教书，还有员工推荐。看起来毫无疑问会很强。

另一个是我在咖啡店认识的人，从来没有做过软件工作，但对机器学习非常感兴趣。

结果第一个人在几周内就离开了，因为不适合我们的文化。第二个人留了下来，后来进入 Shopify，成了非常优秀的机器学习工程师。他在公司圣诞派对上说：“这是我的第一份软件工作。”我们当时都觉得不可思议。

这件事让我更确信：不要让简历替你做判断。

如果我要招最好的赛车手，最有效的办法不是问一堆问题，而是让他上车开。

工作也是一样。我们当然还需要面试，但我们会尽量让面试接近真实工作。入职后的 30、60、90 天同样关键。我们要判断这个人在 Shopify 的真实环境里是否能高影响力地工作。

这对双方都好。如果不适合，越早知道越好。这个人可能在别的团队、别的角色、别的公司非常出色。我们不是要招几百万人，只是想成为世界上最好的万人公司。

## 简历告诉你“做过什么”，但不告诉你“为什么”

Shopify 的招聘里有一个环节叫 life story，也就是人生故事。

我很喜欢一句话：简历的问题是，它告诉你做过什么，却不告诉你为什么做这些事。

一个人从 Stripe 到 Microsoft，重要的不只是公司名字，而是为什么做这个选择。是什么驱动他？他在寻找什么？他是否好奇？是否有足够的跨度？

David Epstein 写过一本书叫《范围》（Range）。这本书对我很难读，因为我一直以为自己是专才。结果书里的数据不断说明，很多复杂领域里，通才反而更有优势。

我后来意识到，我自己其实就是通才。虽然我是工程师，但我也做招聘、HR、薪酬系统设计。比如 Shopify 的薪酬系统重设计，就是我参与完成的。

所以在招聘中，我们会关注一个人是不是有好奇心、有跨度、能不能在变化很快的环境里学习。

入职后，我们也会通过 Slack 发调查，问招聘负责人：你对这个人目前的表现有多满意？这不是为了背后打分，而是为了及时给反馈，帮助双方判断是否匹配。

## 为什么 Shopify 2025 年要招一千名实习生

我非常喜欢 job trial（工作试用）的思路。实习项目其实是最好的工作试用。

2025 年，Shopify 计划招聘一千名实习生。这意味着我们会得到一千个真实工作样本，然后从中判断哪些人适合长期加入。

有些公司把实习项目看作社区服务：支持年轻人，培养早期人才。当然这很好。但我更直接地看：你是说，我可以招一千个带着大脑和 LLM 一起来工作的人？

这一代实习生是在 AI 时代长大的。他们对购物、AI、机器人、聊天、语音有完全不同的体验。他们会给我们带来真实价值。我们也会教他们 Shopify 的工作方式。

如果他们离开，也会带着 Shopify 的印记去别处，这是好事。如果他们留下，就可能成为长期有影响力的人。

我发过这条消息后，一些 CTO 跟我说：那我们要招 1001 个实习生，超过你们。

我觉得很好。如果这能让过去几年被裁员和冻结招聘影响的早期人才生态重新启动，对所有人都是好事。

这里还有一个结对编程里的小秘密：实习生通常比正式员工更有强度。他们的能量会带动飞轮。

我的这个想法和 Waterloo 的 co-op 体系有关。我在 Waterloo 读书时，四个月上学、四个月工作，交替进行。我做了六个四个月的工作期，毕业时不仅有学位，还有两年工作经验。

Shopify 的实习项目也借鉴了这种思路。

但早期人才和资深员工不同。资深员工经历过办公室、远程、跨团队合作，知道怎么处理复杂沟通。很多实习生可能从没真正工作过。所以我们要求实习生每周三天到办公室，目前主要在蒙特利尔、多伦多和渥太华。导师或经理至少每月和他们线下见一次。

这不是和远程文化矛盾，而是尊重早期人才的成长需要。

## 在餐厅里发现人才

我还有一个招聘故事。

有一次我和妻子在餐厅吃饭，我注意到一位服务员。她在很多桌之间快速奔走，脸上一直带着笑，记得住订单，协调厨房，把一个非常忙乱的餐厅组织得井井有条。

我问她：你除了做这个，还做什么？

她说：我就是服务员。

我说：你想不想来 Xtreme Labs 工作？

她一开始不知道那是什么。后来她来了公司，先做前台，然后做我的行政助理，再后来成为招聘人员。我教她招聘，她和我一起去校园宣讲。她非常聪明，只是之前没有机会。

更酷的是，她后来接管了我们的一块 HR 职能。她原本有 college 学历，后来凭借这段经历又补了一年，拿到大学学位。现在她已经是一家公司的 HR director。

这不是孤例。我们后来还从那家餐厅招过更多人。

我一直相信，每个人都在某件事上聪明。关键是你能不能看见这种能力，并判断它能否迁移到新的环境。

## 120 个直接下属，以及为什么我后来改了

在 Xtreme Labs，我曾经有 120 个直接下属。

听起来像噩梦，但它不是一开始就设计成这样的。公司从十个人开始，我让所有人直接向我汇报。随着团队扩大，这个结构延续了下来。

我当时试图验证一件事：员工找经理，通常是为了解决哪些问题？这些问题能不能用系统替代？

“我该做什么？”可以用产品 backlog 回答。

“我在技术上卡住了。”可以通过结对编程快速解决。

“我不知道方向对不对。”可以通过每周内部 demo 和客户 demo 获得反馈。

“我被某个资源卡住了。”这确实需要有人解锁。

所以我不做固定一对一。120 个直接下属，如果每周固定一对一，我什么都不用做了。

我做的是非预约一对一。我会坐在工程师所在楼层中央的一个圆形桌子旁。因为没有太多会议，别人随时可以走过来找我。

线下结对编程还有一个好处：你一眼就能看出是不是卡住了。如果两个人都在电脑前高度投入，说明在运转。如果一个人身体后仰、状态游离，可能就有问题，我可以直接过去问。

后来 Chamath 成为我们的主要投资人。他没有直接说“这行不通”，因为那样我一定会进入防御模式，解释为什么它能行。

他只问我：这套方式到 400 人还能行吗？

我说：可能不行。

他说：那就改。

于是我们加了一点结构。我让几个人成为 director，但仍然要求他们每个人有大约 40 个直接下属，保持组织尽量扁平。

今天在 Shopify，工程管理里通常会有 8 到 20 个直接下属的护栏。有人多一些，有人少一些。但总体原则仍然是：尽量扁平。层级越深，离 Tobi 越远，调查结果里越容易看到对齐度下降。

## 我在 Shopify 第一周做错的决定

我也犯过非常大的错。

2019 年我刚加入 Shopify 的第一周，我们正在重建销售点系统（Point of Sale）。今天这个系统承载着数十亿美元的商品交易总额（GMV），但当时我们要用新的用户体验和新的技术平台重做它。

大家问我：应该用 React Native，还是分别用原生平台开发？

我来自移动开发背景，所以花了很多时间评估。最后我给了一个折中方案：iOS 用 Swift，Android 用 React Native。

我的理由听起来很合理：我们想学习 React Native，而 Android 是更难的平台；同时 iOS 用 Swift 可以保证至少有一个产品顺利上市。

一年后，iOS 版本发布，很成功。然后我们又花了大约六个月做 Android 的 React Native 版本。

很快我们意识到：React Native 才是未来。它能让我们拥有一个统一平台，也能让 Web 侧熟悉 React 的工程师参与移动开发。后来 Shop app 也用了 React Native，我们对它的判断越来越清楚。

于是我不得不回去告诉大家：我犯了大错。我们已经花一年做出的 iOS 版本，需要重新基于 React Native 改写。

我估算过，这个决定大概烧掉了 100 名工程师 18 个月的时间。

这是我加入 Shopify 第一周做出的决定。

我去找 Tobi，说我犯了这个错误。Tobi 没有骂我。他只说：你应该把这个故事告诉所有人。

然后他问我：你知道自己错在哪里吗？

我一开始以为是在技术判断上错了。但他说：你的问题是没有承担风险。我会严厉对待不承担风险的人。如果你承担了正确的风险，最后没成功，你不会因此惹上麻烦。但这一次，你没有冒那个险。

他是对的。

我当时知道自己会全力投入 React Native。我后来飞去 Meta 见 React 团队，成为 React Native 核心贡献者，运行 React Native working group，做 release captain。既然我知道自己会把所有事都做到这个程度，我就应该判断：如果我全力以赴，这条路真的会失败吗？

我没有这样的信心，所以选择了折中。

折中有时是最糟糕的选择。

今天 Shopify 的相关移动应用已经全面转向 React Native，我们也是核心贡献者。但那次失败让我记住一件事：难路不是盲目冒险，而是在你已经知道自己会全力投入时，不要因为害怕显得激进而退回中间地带。

## 我常用的几个提醒

如果只能留下几句经验，我会这样说。

不要把强度理解成加班。真正的强度，是在同样的时间里提高学习密度、反馈密度和决策密度。

不要迷信会议少。无意义会议应该删掉，但高质量检查点能让组织更快纠偏。关键不是开不开会，而是会议有没有让更好的上下文进入问题。

不要迷信代码多。代码越多，组织越慢。能删除代码、简化系统、构建基础设施，才是真正的长期加速。

不要只用面试判断人。尽量让候选人进入真实工作，让工作样本说话。

不要害怕看起来很笨。很多突破都来自一句别人不敢问的问题。

我很喜欢的一句座右铭是：Everything you know is wrong。你知道的一切都可能是错的。

它不是虚无主义，而是一种第一性原理的提醒：如果现有知识都不可靠，你能不能从事实和实验重新搭建对世界的理解？

这也是我喜欢不断尝试的原因。很多事你不试，就永远不知道边界在哪里。

在 Shopify，我们也是这样工作：选更难但更能学习的路，保持高密度节奏，删除无意义的会议和代码，搭建让别人更快的基础设施，用真实工作发现真实人才。

高强度文化不是让人筋疲力尽。它真正要做的，是让每个人都更快接近事实、更快学到东西、更快把有价值的产品交到用户手里。
