# 2025-04-13-v0 的野心：让一亿人成为软件创造者

昨天，有个用户给我们留下了一条反馈。

他说，v0 像一个“超级天才、五岁、博士毕业、还带点注意力不稳定”的小孩。

我觉得这个形容很准确。我不想把它吹得太神。它不会每次都给出完美答案，也会犯错，也会跑偏。但它确实已经有很多让人惊讶的瞬间：它好像什么都懂一点，偶尔还会在某个细节上表现得超过人类专家。

我现在最关心的问题不是“AI 能不能写代码”。这个问题已经不新鲜了。真正重要的问题是：当一个产品经理、设计师、营销同事、销售工程师，甚至一个普通创业者，都能把想法直接变成可运行的软件时，产品开发会变成什么样？

我的答案是：每个人都会更像工程师。

不是说每个人都要变成传统意义上的程序员，而是每个人都能更直接地参与“构建”。过去很多想法会停在文档、会议、截图、Figma 草图里。现在，我希望它们能变成链接，变成原型，变成可交互的界面，甚至直接变成线上产品。

这就是我们做 v0 的原因。

## 我一直在做的事：把难事变简单

我创办 Vercel 已经快十年了。更早之前，我就在做开发者工具。

很多人认识 Vercel，是因为 Next.js。它是一个基于 React 的开源框架。React 来自 Meta，Next.js 则被很多互联网上最前沿的产品使用。你今天用 Claude、Grok、Midjourney 这些产品时，很可能就在使用 Next.js 和 Vercel 的技术。

在 Next.js 之前，我还做过 Socket.IO。它解决的是实时通信问题。比如 Notion 里多人协作时，消息要实时同步到其他协作者那里，这背后就有类似的实时通信引擎。

回头看，我做的事情其实一直很一致：把一件很难、但很有价值的事情，变得尽可能容易。

实时通信很难，我做 Socket.IO。构建现代 Web 应用很难，我们做 Next.js 和 Vercel。可即便如此，过去你仍然需要会开发，才能真正把东西做出来。

到了 v0，这个边界开始变了。

我粗略算过，全球可能有 500 万 React 开发者，2000 万 JavaScript 开发者。但如果问“有多少人想构建产品”，这个数字至少是一亿。

我这个数字不是精确统计，而是从一个很现实的现象来的：Slack 大约有一亿月活用户。很多人在 Slack 里讨论产品、客户、功能、想法。他们会说：“我想要这个功能”“我希望能看到那种界面”“客户需要这样的流程”。

过去，这些话只是讨论。现在的问题变成了：如果你能直接对电脑说出这个想法，然后看到一个东西被生成出来呢？

一个原型。一个 demo。一个全栈（full stack）应用。甚至一个能上线的产品。

这就是 v0 的机会。

## v0 为什么会在这个时间点出现

v0 的诞生，和 ChatGPT 出现的时间点密切相关。

ChatGPT 刚出来时，我们很快发现，它非常擅长写我们这套工具链里的代码：JavaScript、Tailwind、Next.js。这其实是开源长期主义的回报。因为这些工具早就在互联网上有大量公开资料，模型训练时已经见过它们。

最早我们从 OpenAI 的模型开始。后来，我们一直会组合使用不同模型。你可以把它理解成一条模型流水线：有的模型擅长这个任务，有的模型擅长另一个任务，一个模型会把工作交给另一个模型。

OpenAI、Gemini、Claude，我们都会看。但 v0 的第一版想法甚至可能早于 ChatGPT，至少当时我们已经在 GPT-3.5 这一代能力上探索了。

为什么大语言模型（LLM，Large Language Model）特别适合 Web 设计和 Web 开发？

因为很多开发工作，本质上是“翻译”。

把一张 Figma 设计图翻译成 React 组件。把一段产品意图翻译成界面。把“这个按钮要更醒目一点”翻译成 CSS、布局、边距、圆角、颜色和交互状态。

过去，这些翻译工作需要很多专业技能。比如前端里有人非常擅长 CSS 和 Tailwind，知道盒模型、flex 布局、padding、margin、border 的每个细节。现在，v0 已经非常擅长这类工作。

我会经常拿自己的个人网站做测试。以前，我用自己创造的 Next.js 去写它，花了很长时间。上一代模型要复刻它，大概需要 10 到 15 次提示词。最近这一代模型，只需要两次。

更让我谦卑的是，有一次 v0 生成的代码在可访问性（accessibility）上比我自己写得更好。（**可访问性指网站对不同用户都友好，例如读屏软件、键盘操作、颜色对比度等。**）因为模型见过大量 Web 标准，它会自动遵循很多规范。

所以，我不认为所有工程工作都会消失。但很多过去被当成专业工种的“翻译任务”，会被 AI 大幅改变。

## 工程师不会消失，但“只会翻译”的工作会减少

很多人问我：以后还需要工程师吗？我还应该学代码吗？

我的回答是：理解事物如何运转，仍然是世界上最重要的能力之一。

AI 会让一个人做过去一百个不同专业的人才能完成的事。一个设计师可以发布完整产品，一个产品经理可以把原型做成线上版本，一个后端工程师可以把自己的 API 和基础设施想法做成端到端的产品体验。

但越是这样，越需要知道底层发生了什么。

你不一定要记住每一个 CSS 属性，也不一定要像过去的前端专家一样追踪每个浏览器何时支持某个规范。未来这类“百科全书式”的记忆会变得没那么重要。

真正重要的是，你要知道有哪些概念，知道该怎么描述它们。

你要知道 CSS、布局、组件（component）、数据库、API、上下文窗口（context window）这些词分别意味着什么。因为当你知道这些“符号系统”时，你就能更好地影响模型，让它更接近你的意图。

我有五个孩子。如果他们问我，未来应该学什么，我会很明确地说：数学、表达能力、展示能力，以及对软件如何运转的理解。

数学训练的是底层逻辑。表达能力让你能准确指挥 AI，也能把你的想法讲给别人。展示能力会变得越来越重要，因为软件的边际生产成本正在下降，世界上会出现越来越多产品。你不仅要会做，还要会告诉别人你做了什么。

至于编程，我仍然鼓励学习。尤其是 JavaScript。它某种程度上像编程世界里的英语：学会它，你就能触达数十亿设备。ChatGPT、Claude、Gemini 在生成 Web 应用时，常常会选择 JavaScript、React、Next.js，这不是巧合。它已经成了构建 Web 产品的通用语言。

不过，真正长期有价值的工程能力，是基础设施能力。

大语言模型可以写很多代码，但它不是从零创造整个宇宙。它会调用已有框架、编译器、云服务、数据库和基础设施。今天的 AI 还很难自己从零写一个云平台、一个编译器、一个完整框架。能构建这些底层系统的工程师，未来很多年仍然会非常重要。

## v0 想做的不是“写代码工具”，而是“社会化产品构建”

现在，v0 已经有超过 130 万用户使用过。我们最近还经历了上线以来最大的一天，甚至已经成为很多云服务商的重要客户，因为 GPU 和 LLM 基础设施都被我们推到了很高负载。

但最让我兴奋的不是数字，而是人们用 v0 做了什么。

我们上线了 v0 Community。不到一个月，就有 2 万多个社区提交。有些作品被 fork 了 1200 次、1500 次、6000 次。

这让我想到 GitHub。

GitHub 最早的标语是“social coding”，社会化编程。它让开发者可以分享代码、fork 项目、在别人基础上继续构建。它极大地推动了软件民主化。

但 GitHub 仍然要求你会写代码。

v0 想做的是下一步：social product building，社会化产品构建。

一个人做出一个应用，另一个人可以拿来改。你不必从空白页面开始，可以从别人已经做出的东西出发，再用自己的提示词把它变成你想要的样子。

这也是为什么我觉得 v0 颠倒了 Git commit 的流程。

传统工程师是先在编辑器里工作很久，最后提交一个 Git commit，用一句话总结“我刚才做了什么”。v0 是反过来的：你先说出意图，比如“把这个按钮改成红色，点击后把表单保存到数据库”，然后系统生成代码。

意图在前，代码在后。

这会改变产品团队的协作方式。

## 未来的产品讨论，会从文档变成可运行的链接

过去，产品经理和客户之间的讨论，经常停留在语言和文档层面。大家会说需求，写 PRD（Product Requirements Document，产品需求文档），做设计稿，再排工程资源。

但我认为，未来这些讨论会越来越多地被“可运行的产物”承载。

Claude 把类似的东西叫 artifacts，我很喜欢这个词。因为我们不是只想聊天，我们是想把想法做出来，分享给世界。

在公司里，部门之间常常会互相等待：市场要设计支持，设计要工程实现，工程又要更清楚的产品说明。每个环节都会有距离感。你明明知道自己想改哪里，却不能直接改。

而 v0 想消除这种无力感。

我们曾在 Vercel 博客里写过设计工程（design engineering）。这类人同时有设计和工程能力，在 Vercel 内部非常有效。v0 的一个重要灵感就是：也许更多人都可以变得更全栈。

产品经理不只是写需求，而是能做出交互版本。设计师不只是交付视觉稿，而是能发布完整产品。后端工程师不只是交付 API，而是能把自己的系统能力包装成用户真正能触摸到的体验。

Vercel 有 600 名员工，其中大约 150 名是传统意义上的工程师。但从某种意义上说，有了 AI 以后，我们现在有 600 名“工程师”。

我见过市场团队用 v0 做基础设施 demo，比静态图和落地页更清楚。我见过设计师用视频模型生成大会动画。我见过销售工程副总裁做出帮助潜在客户理解 Vercel 的工具。我们每周五有 Demo Friday，让大家展示自己做出来的东西。这个仪式很重要，因为它给了非工程团队“你也可以构建和发布”的许可。

## 如何更好地使用 v0：大胆、迭代、会求助

如果我坐在一个第一次使用 v0 的人旁边，我会给他几个建议。

第一，要足够大胆。不要一开始就把要求缩得很小。你可以直接说：“帮我做世界上最好的飞行雷达。”

这不是比喻。我真的这么做过。

有一次我从东京飞往旧金山，机上网络非常差。我喜欢在飞行中看自己的航班位置，但当时用到的飞行雷达网站太臃肿，有广告，也不是我想要的样子。我就在飞机上用 v0 做了一个自己的飞行雷达。

我没有告诉它具体用什么技术。v0 自己选了 Mapbox 和 Leaflet。后来我让它接入 Edge Aviation 的航班数据 API。再后来，我发现同时渲染的航班太多，性能会有压力。我只是告诉它：“现在航班太多了，我们得优化一下。”它最后做出了基于 canvas 的覆盖层，能在地图上渲染成千上万架航班。

整个过程不到两个小时，用的还是飞机上很差的网络。我付的是每月 20 美元的 v0 订阅。如果让工程师完全不用 AI 来做，可能要几周。哪怕 Vercel 最强的工程师，也许能几天做出来，但设计呢？产品判断呢？很多时候，瓶颈不是工程师，而是我能不能把自己想要的体验讲清楚。

第二，要给它灵感。你可以从 v0 Community fork 一个作品开始。你也可以上传截图、Figma 文件，或者告诉它一种风格：新粗野主义（neobrutalist）、极简、报纸风、复古、像 Telegram 一样。

第三，要像指导一个合作伙伴一样迭代。有时候一句“换一种方式试试”就足够让它走出卡住的地方。很多用户会点反馈按钮告诉我们“我本来想让 v0 做什么”，但在很多情况下，我们想说的是：你直接把这句话告诉 v0 就好了。

第四，要学会“脱困”。

React 有一个很重要的设计思想，叫逃生口（escape hatch）。（**意思是当框架的默认抽象不够用时，给开发者留一个可以绕出去、直接处理底层问题的出口。**）v0 也有这个能力：你能看到它生成的代码，能编辑它，能把代码拿给别的专家或别的 AI 看。

Luma Labs 有位设计工程师用 v0 做了一个很复杂的动画项目，迭代了大约 120 次。后来他卡住了，就把 v0 生成的代码复制给 ChatGPT o1，让 o1 帮他思考解决方案。这就是未来的工作方式之一：你不只是向一个 AI 求助，而是在不同工具之间调度能力。

## AI 产品的核心优势，是反馈循环更紧

做 AI 产品让我学到一件事：反馈会变得极其重要。

早期 Vercel 的一个重要做法，是在产品里放一个反馈按钮。用户可以用几个表情表达当下感受，反馈会直接进 Slack。我们每天都能看到用户真正的想法。

到了 AI 产品，这个反馈流更密集。用户会点赞、点踩、指出模型在哪里理解错了。下一轮模型、提示词、微调、示例、RAG（Retrieval-Augmented Generation，检索增强生成）都可以因此变得更好。（**RAG 是让模型在回答或生成前检索相关资料，从而减少胡编和过时信息的一类方法。**）

所以，如果你在做产品，我非常建议你在产品内部创造大量反馈入口。不要只等用户发邮件，也不要只等访谈。让反馈自然地出现在使用过程中。

我们也开源了很多 v0 背后的基础能力。比如 Vercel 的 AI SDK，它是 v0 的底层管道之一，可以连接不同模型，不只是生成文字，还可以生成 UI。我们还提供 ChatGPT 风格的开源模板，让开发者可以更容易地做自己的 AI 产品。

这会催生很多垂直 AI 工具。

我相信专家型 AI 工具会很有前途。比如 ChatPRD 是面向 PRD 写作的 v0。OpenEvidence 像医生使用的 ChatGPT。还有做 X 光 AI 工具的创业公司。我们法务团队喜欢 GetGC.AI，因为它专注法律场景，创始人本人也是律师。

从理论上讲，未来足够强的 AI 也许可以实时生成一个法律工具。但人类仍然需要护栏、方向和判断。AI 现在仍然更像副驾驶、伙伴、助手。它并不总知道我们真正想要什么，也不一定知道什么对我们最好。

## 品味不是天生的，是暴露时间训练出来的

现在大家都在谈“品味”。好像未来最重要的能力，就是知道什么是好产品、什么是好设计、什么体验值得用户喜欢。

我同意品味重要，但我不认为它是天生的。

在 Vercel，我们有一个内部原则：增加暴露时间。

也就是尽量量化你花了多少时间观察用户使用你的产品，花了多少时间体验别人的优秀产品，花了多少时间把自己的东西拿出去给真实用户看。

品味就是这样练出来的。

你要多用产品。用最好的产品。观察它们的小细节。比如我最近做过一个受 Grok 启发的 ChatGPT 风格界面。我注意到 Grok 在移动网页上按回车默认换行，因为手机用户通常不是用回车发送消息；但在桌面端，按回车默认发送，因为这是桌面用户的预期。

这种细节很小，但它决定了体验是否顺手。

另一个训练品味的方法，是把产品交给别人，当面看他怎么用。你会很痛苦，因为你会发现很多自己以为没问题的地方，其实别人根本用不明白。但这种痛苦非常有价值。你越接近真实世界，越能建立判断力。

我会让团队给我的日历上色，确保我有足够多的一对一会议，也有足够多的客户会议。在客户会议里，我会尽量亲自使用客户的产品，观察他们怎么成功、怎么卡住。有时一周三分之一的会议可能都和客户有关。

我们也会邀请客户现场演示他们如何使用产品。每次都会发现一些我们不知道的痛点，或者发现某个我们以为很直观的功能，其实并不直观。

这就是暴露时间。

## v0 仍然有限制，但边界在快速移动

v0 现在已经能通过 zip 文件导入代码库，Git 集成也很快会到来。它可以做全栈开发，可以连接 API。我们也在做更紧密的 Vercel Marketplace 集成：当你需要数据库、AI 模型，或者 v0 判断它自己需要这些基础设施时，它可以从市场里无缝安装。

但我不会说它已经完美。

AI 仍然会犯错。你可能会遇到运行时错误。我们也在产品输入框下方提醒用户这一点。好消息是，v0 越来越像智能体（agentic），也就是说它遇到错误时，会尝试自己修复。

另一个限制是大型代码库。今天的 AI 对特别庞大的代码库仍然没那么擅长。不过，React 和 Next.js 的组件化结构天然适合模型处理。你可以把任务缩小到某个组件、某个文件、某个页面，减少模型在超长上下文里迷路的概率。

这也是为什么我一直强调：不要只把 v0 当成“自动写代码”。更好的方式是“产品优先，代码最后”。你生活在产品里，直接看界面、调体验、改状态。代码仍然存在，也很重要，但它不再是每个人工作的第一入口。

## 最后，产品质量仍然来自细节

AI 会让构建速度快很多，但它不会替代质量意识。

一个伟大的产品由上千个细节构成。你永远不会真正完成。优秀的产品建设者往往会对九个想法说不，只对一个想法说是。因为每个“是”都会带来长期维护成本。

所以，速度变快以后，克制更重要。

你要不断自用产品，不断测试，不断观察真实用户，不断修正。Vercel 和 v0 都是我们自己每天大量使用的工具。我们用它们来构建 Vercel，也用它们来构建 v0。

我期待的未来很简单：如果你能想到，你就能发布。

当更多人能把自己的想法做出来，软件会变得更快、更高质量、更少错误。我们最终也许会停止反复谈“AI”，因为 AI 会变成软件的一部分。我们构建软件，也使用软件来构建软件。

到那时，真正稀缺的不是“谁会敲代码”，而是谁更理解问题，谁更懂用户，谁更会表达意图，谁更有品味，谁更能把一个想法推到真实世界里。

这就是 v0 想参与的未来。
