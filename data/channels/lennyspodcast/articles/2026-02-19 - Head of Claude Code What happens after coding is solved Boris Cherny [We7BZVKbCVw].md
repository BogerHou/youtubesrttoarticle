# 2026-02-19-Claude Code负责人：当写代码被AI解决之后

从 2025 年 11 月开始，我没有再手写过一行代码。

这句话听起来很夸张，但它就是我现在的日常。我的代码 100% 由 Claude Code 写出来。每天我会发 10 个、20 个，有时甚至 30 个 pull request（代码合并请求）。我仍然会看代码，仍然会确认它是否正确、安全、符合产品方向，但敲下每一行代码这件事，已经不再由我完成。

更有意思的是，我并没有因此变得不喜欢编程。恰恰相反，我从来没有像今天这样享受“写代码”。

因为今天的写代码，已经不再只是盯着语法、依赖、Git、测试报错和各种琐碎细节。今天的写代码，更像是在表达意图、判断方向、拆解问题，然后让一个足够强的 AI 编程代理（coding agent）去完成具体实现。

我负责 Anthropic 的 Claude Code。回头看过去这一年，它对软件工程的改变速度，连我自己都觉得不真实。

有报告说，现在 GitHub 上所有公开提交里，已经有 4% 是由 Claude Code 完成的，而且预测到今年年底可能会接近五分之一。这个数字还只是公开仓库。如果算上企业内部的私有仓库，我相信比例会更高。更疯狂的是，增长并没有放缓，反而还在继续加速。

很多人会问：这到底是怎么发生的？Claude Code 一开始就是一个宏大的产品计划吗？

不是。

它一开始只是一个小 hack。

## 我为什么又回到了 Anthropic

在讲 Claude Code 之前，我想先讲一段小插曲。大约半年前，我确实离开过 Anthropic，去了 Cursor。但两周之后，我又回来了。

这可能是我职业生涯里最快的一次换工作。

我去 Cursor，是因为我真的很喜欢那个产品，也很喜欢那个团队。他们很早就看到了 AI 编程的方向，做出了非常酷的东西。对一个产品工程师来说，参与那样的产品当然很有吸引力。

但我很快意识到，我真正想念 Anthropic 的，是它的使命。

在加入 Anthropic 之前，我在大公司工作过。后来我越来越想去一家 AI 实验室，因为我希望能以某种方式参与塑造这项技术的未来。而 Anthropic 吸引我的核心原因，就是它对安全的重视。

在 Anthropic，你随便在走廊里拉住一个人，问他为什么在这里，答案几乎总是同一个：安全。

这种使命感对我非常重要。再酷的产品、再令人兴奋的机会，都替代不了这一点。所以我很快就知道，自己应该回去。

## Claude Code 最早只是一个终端里的原型

我刚加入 Anthropic 时，先花了一个月做各种原型。很多原型很奇怪，也没有上线价值，但它们帮我理解模型的边界。

接着我又花了一个月做后训练（post-training），也就是理解模型在训练之后如何被进一步调整，让它更符合具体任务。作为工程师，我一直相信一件事：你要做好某一层的工作，就必须理解它下面那一层。

传统软件工程里，如果你做产品，最好理解基础设施、运行时、虚拟机、编程语言这些底层东西。做 AI 产品也一样，你必须在某种程度上理解模型本身。

后来我开始做一个原型，当时它叫 ClaudeCLI。最早的版本非常简单，运行在终端（terminal）里，因为那是最快、最省事的方式。那时团队几乎只有我一个人，没有复杂界面，也没有完整产品设计。

有一次，我给模型提供了一个 bash 工具，然后问它：“我现在在听什么音乐？”

我并没有告诉它应该怎么做。没有写一堆规则，也没有说第一步该调用哪个命令、第二步该看哪个文件。模型只是拿到了工具，然后自己想办法去回答这个问题。它真的写代码、调用系统信息，找出了答案。

那一刻我很震惊。因为这意味着模型不只是会聊天，也不只是会补全代码。它开始会使用工具（tool use）了。（**工具使用指的是模型能调用命令、API、文件系统、浏览器等外部能力，自己获取信息并采取行动，而不是只在对话框里生成文字。**）

我把这个原型发到公司内部，结果只收到了两个赞。

这并不奇怪。大家一想到编程工具，脑子里通常是 IDE，是复杂的编辑器，是很成熟的工程环境。没有多少人相信，一个终端里的小工具会变成重要产品。

但后来我越来越觉得，终端反而是优势。

因为模型变化太快了。任何精致的界面都可能很快跟不上模型能力。终端很原始，但它足够灵活，能让模型自己决定做什么、怎么做、以什么顺序做。

这也是我后来总结出的第一个产品经验：早期要适当资源不足。

不是说要故意让团队痛苦，而是当资源不那么充裕时，你会被迫找到最直接、最有杠杆的做法。Claude Code 如果一开始就有一个大团队，很可能会被做成一个传统 IDE 插件。但它最早只有一个人，所以它被做成了终端里的代理。

这个限制，反而让它保留了模型最强的部分。

## 它不是一发布就爆了

Claude Code 在 Anthropic 内部最先火起来。

早期，Ben Mann 建议我做一个 DAU（日活用户）图表。我当时还觉得太早了，但他坚持让我做。结果图表很快就垂直向上。

2025 年 2 月，我们把 Claude Code 对外发布。但很多人可能忘了，它一开始并不是那种瞬间爆红的产品。

确实有一批早期用户马上理解了它，也有不少人开始使用。但对大多数人来说，它太陌生了。一个终端里的 AI 编程代理，到底该怎么用？它和聊天机器人有什么不同？它为什么不是传统 IDE？

这些问题都需要时间。

后来 Claude Code 逐渐出现在更多地方：iOS 和 Android 的 Claude app、桌面应用、网页、IDE 扩展、Slack、GitHub。它不再只属于终端用户，而是慢慢进入工程师已经工作的地方。

这背后有一个很重要的产品原则：潜在需求（latent demand）。

潜在需求指的是，用户已经在用一个产品做它原本没有设计的事情。产品团队如果看懂这种“误用”，就能发现下一步应该做什么。

Facebook Marketplace 是一个经典例子。早年 Facebook 发现，群组里有很大比例的帖子都在买卖东西。群组本来不是为交易设计的，但用户硬是拿它做交易。这说明需求已经存在，只是产品还没承接它。

Claude Code 也是这样。

我们看到很多人用它做非编程任务：有人用它种番茄，有人用它分析基因组，有人用它从损坏硬盘里恢复婚礼照片，还有人用它分析 MRI 图像。我们内部的数据科学家 Brendan 甚至自己打开终端，装 Node.js，装 Claude Code，然后在终端里做 SQL 分析。很快，其他数据科学家也都开始这么做。

这太明显了。人们愿意费劲打开终端，只为了让 AI 做这些事，那就说明我们应该给他们做一个更自然的产品。

这就是后来 co-work 产品的来源。团队用了大约 10 天，把 Claude Code 的能力放进桌面应用里，做成一个更适合非工程师使用的代理。它不是简单的聊天框，里面还包含很复杂的安全系统，比如虚拟机和护栏，确保模型在执行任务时不会随便越界。

## 当 100% 的代码由 AI 写出，工程师还做什么

我现在的工作流非常简单：我描述目标，Claude Code 写代码。

但这不等于我完全不看代码。

在 Anthropic，Claude 会自动审查 100% 的 pull request，之后还有人工审查。尤其是会进入生产环境的代码，人仍然要负责判断正确性、安全性和产品影响。

所以我不认为今天已经到了“完全放手”的阶段。至少在严肃工程里，人类检查点依然重要。

但趋势已经非常清楚了：AI 正在承担越来越多的具体实现工作。

我在 2025 年 5 月参加 Anthropic 的开发者会议时，有人问我年底会发生什么。我当时说，到年底，工程师可能不再需要 IDE 写代码。全场能听到明显的惊讶声，因为这听起来太疯狂了。

但在 Anthropic，我们很习惯用指数曲线思考问题。我们公司的几位联合创始人，本身就是缩放定律（scaling laws）论文的重要作者。（**缩放定律指的是，在一定范围内，模型规模、数据量和计算量增加时，AI 能力会呈现相对可预测的提升趋势。**）

当时如果你把 Claude Code 写代码比例的增长曲线拉出来看，年底接近 100% 并不是不可能。它只是不符合直觉。

到 11 月，这件事真的发生在我自己身上。从那之后，我没有再手写代码。

Claude Code 发布早期，大概只写我 20% 的代码。到 5 月，也许是 30%。我那时仍然主要用 Cursor。真正的拐点来自更强的模型，比如 Opus 4 和 Sonnet 4。模型一强，Claude Code 的价值就立刻被放大。

这给 AI 产品创业者一个很重要的启发：不要只为今天的模型做产品，要为 6 个月后的模型做产品。

这会很难受，因为前 6 个月你的产品市场契合度可能没那么好。但如果你押对方向，等模型能力上来，产品会突然“咔哒”一声对上。

Claude Code 就是这样。

## 下一步不是写代码，而是决定写什么

当 AI 已经能写出大部分代码之后，下一个瓶颈是什么？

不是实现，而是判断。

现在 Claude 已经开始帮助我们想“应该做什么”。它会看用户反馈、bug 报告、遥测数据，然后提出可以修复的问题、可以上线的改进，甚至直接开 pull request。

这让它越来越像一个同事。

早期我会自己盯着内部反馈频道。只要有人反馈 Claude Code 的问题，我就尽量在 1 分钟、5 分钟内修掉。这个反馈循环非常重要。用户如果发现自己的反馈马上被听见、马上被解决，就会更愿意继续反馈。

现在很多这类工作可以交给 Claude。你可以把它指向一个 Slack 线程，让它读完讨论，列出可以行动的事项，再开几个 PR 给你看。

这对产品经理当然很有冲击。

因为过去我们会说，AI 先解决写代码，接下来人类还负责决定做什么。但现在，AI 也开始进入这个区域：从反馈里找模式，从 bug 里找机会，从用户对话里提出改进。

我认为到今年年底，工程、产品、设计这些边界会变得更模糊。软件工程师这个头衔，可能会在一些团队里慢慢被 builder 取代。也可能变成另一种状态：每个人都是产品经理，每个人也都会写代码。

在 Claude Code 团队，已经是这样了。

我们的产品经理会写代码，工程经理会写代码，设计师会写代码，财务同事会写代码，数据科学家也会写代码。并不是每个人都做同样的事，但很多角色之间已经有大约一半工作在重叠。

未来最有优势的人，可能不是某个狭窄领域的专家，而是 AI 原生的通才。他懂工程，也懂产品；懂设计，也懂用户；懂业务，也能自己动手把东西做出来。

## 我不怀念手写代码

有人问我：你会怀念亲手写代码吗？

我的答案是：不太会。

我学编程最早是非常功利的。上中学时，我为了数学考试“偷懒”，把答案写进 TI-83 Plus 图形计算器。后来题目变复杂，没法提前写答案，我就写了一个小求解器。再后来我发现可以买一根线，把程序传给全班同学，于是全班都拿了高分。最后当然被老师发现了，让我们别再这么干。

从那时起，编程对我来说就是一种工具。它的意义是让我能做出东西。

当然，我后来也沉迷过编程本身的美感。我写过 TypeScript 的书，也组织过当时世界上最大的 TypeScript meetup。我喜欢函数式编程（functional programming），喜欢类型系统（type system），喜欢那种把复杂逻辑整理得漂亮、严密、像解数学题一样的感觉。

但即使如此，写代码本身也不是终点。终点是构建东西。

所以当 Claude Code 帮我处理掉大量琐碎细节之后，我反而更享受工作了。因为我能把更多时间放在真正有趣的部分：理解用户、判断方向、设计系统、和团队讨论未来。

当然，并不是每个人都这样。我们团队里也有人周末还会手写 C++，因为她就是享受手写 C++。未来依然会有人把编程当作手艺和艺术来享受，这完全没问题。

但对我来说，编程一直在变化。

最早的“计算”是人拿纸笔做数学。后来是硬件、开关、打孔卡（punch card），再后来是软件、虚拟机、现代编程语言。每一次变化，都会有人说“这不是真正的编程”。我外公就是苏联早期程序员之一，他用打孔卡编程。对他来说，那就是编程。后来软件出现，上一代程序员也许同样觉得那不够“正宗”。

现在只是又一次变化。

今天你可能还需要理解模型下面那一层，也就是代码、系统和运行机制。但一两年后，这件事可能就没那么重要了。就像今天大多数程序员不需要每天写汇编一样，代码也可能变成 AI 编程代理下面的一层。

## 这更像印刷机，而不是普通工具升级

我一直在想，AI 编程在历史上最像什么？

我想到的类比是印刷机。

在 15 世纪中期的欧洲，识字率很低，可能不到 1%。阅读和书写主要属于抄写员。他们为贵族和国王服务，而很多贵族和国王本人并不识字。

后来古腾堡印刷机出现。接下来的 50 年里，印刷材料的总量超过了之前 1000 年的总和，成本下降了大约 100 倍。识字率没有立刻暴涨，因为教育需要时间，人也需要从农业劳动里腾出学习时间。但在接下来的 200 年里，全球识字率大幅提高。

有一个很有意思的历史记录：有人问当时的抄写员怎么看印刷机。那位抄写员并不沮丧，反而很兴奋。因为他不喜欢的工作是一本书抄到另一本书，喜欢的是给书画插图、做装帧。印刷机把最枯燥的复制工作拿走了，让他能做更有创造性的部分。

我对 Claude Code 的感受很像。

我不怀念那些琐碎工作。真正有意思的是决定做什么、为什么做、怎样把系统搭起来、怎样让用户更好地使用它。

如果未来每个人都能用自然语言构建软件，会发生什么？我不知道。就像 15 世纪的人不可能预测文艺复兴、现代科学和今天的互联网一样，我们也很难预测这个能力普及之后会解锁什么。

乐观的一面是，编程会民主化。任何人都可以在任何时候构建软件。

现实的一面是，这个过程会非常有冲击，也会让很多人痛苦。工作会变化，角色会变化，组织方式会变化。这个问题不应该由某家公司单独决定，而应该由整个社会一起讨论。

## 如果你想跟上，先成为通才

如果一个普通人问我：面对这个变化，我该怎么办？

我的第一条建议很简单：去用这些工具。不要害怕它们。尽量站到前沿，亲手试。

第二条建议是：比过去更努力地成为通才。

很多学计算机的人，学校里主要学写代码，最多再学一点系统架构。但我每天看到的高效人才，往往跨越多个领域。

最强的工程师，可能同时懂产品和基础设施。也可能是产品工程师，但有很好的设计感觉。也可能是工程师，但很懂商业，能判断下一步最该做什么。还有一种人很喜欢和用户聊天，能把用户真正想要的东西转化成产品方向。

未来几年，最被奖励的人，不只是会用 AI，而是能把 AI 用在更大的问题上。他们不只盯着工程部分，而是能理解整个问题。

这也是为什么我认为工程、产品、设计、数据这些角色都会被影响。任何大量使用电脑工具的工作，都会被 AI 代理改变。

这里的“代理”（agent）不是一个营销词，而有明确含义：它是能使用工具的大语言模型。它不只是回答问题，还能操作 Google Docs、发邮件、运行命令、打开浏览器、读文件、填表格，真正替你做事。

这和聊天机器人是完全不同的体验。

很多非工程师今天还没有真正用过代理，所以他们还没有感受到这种变化。但我认为，接下来被冲击最明显的，会是产品经理、设计师、数据科学家，以及所有和工程相邻的岗位。

## 少一点人，反而可能用好 AI

在团队管理上，我有一个看法：早期项目不要太早堆人。

如果你招到很强的工程师，再给他们足够权限和足够 token，他们会想办法完成事情。人少一点，反而会迫使他们更充分地使用 Claude。

我经常给 CTO 的建议是：一开始不要急着优化成本，不要先想着省 token。先给工程师尽可能多的 token，让他们自由实验。

有人可能会说：你在 Anthropic 工作，当然希望大家多用 token。

但我的真实观点是，在小规模探索阶段，token 成本相比工程师薪水和公司其他成本，通常并不算大。你应该先让想法跑起来。等某个想法真的成功，开始规模化，成本变大了，再去优化模型、提示词、流程，甚至考虑用更便宜的模型。

如果太早省钱，很多疯狂但有价值的想法根本不会被试出来。

在 Anthropic，我们已经看到有些工程师每个月花掉几十万美元的 token。这个数字听起来很大，但它背后对应的是一种新的工作方式：一个人同时启动多个代理，让它们并行完成任务。

我自己录节目时，也经常有五个代理在跑。早上醒来第一件事，可能就是打开手机上的 Claude iOS app，让一个代理去检查昨晚写的东西。现在我大概三分之一的代码工作在终端里做，三分之一在桌面应用里做，还有三分之一在手机上做。

这在几年前听起来完全不可思议。但现在，它就是日常。

## 做 AI 产品，不要把模型关进盒子里

如果你正在做 AI 产品，我最想强调的原则是：不要把模型框得太死。

很多人的直觉是，把模型当作大系统里的一个小组件，然后设计一套严格流程：第一步必须做什么，第二步必须做什么，第三步必须做什么。再配一个复杂的编排器（orchestrator），让模型按路线走。

一年前，这样做可能还有必要。但今天，通常更好的方式是给模型工具、给它目标，然后让它自己决定如何完成。

不要一开始塞给它一堆上下文。更好的做法是给它获取上下文的工具，让它自己去读需要的信息。

这和 Rich Sutton 提出的“苦涩教训”（Bitter Lesson）有关。这个观点大意是：长期看，更通用的方法往往会胜过更特化的方法。（**在 AI 领域，很多靠人工规则和领域技巧堆出来的短期优势，最终会被更大规模、更通用的学习系统超过。**）

我在 Claude Code 上也看到类似现象。复杂脚手架可能让当前模型提升 10% 或 20%，但下一代模型一出来，这些收益常常就被抹平。长期来看，押注更通用的模型，往往更可靠。

所以我们当初做 Claude Code 时，选择把产品设计成“模型本身就是产品”。我们只加最少的脚手架，提供最必要的工具，然后尽量让模型自己发挥。

这在当时很冒险，因为早期模型还不够强。但我们押的是 6 个月后的模型。

现在看，这是正确选择。

## 安全不是上线之后再想的事

在 Anthropic，安全不是产品上线之后才补的东西。

我们看模型安全，大致有三层。

第一层是对齐（alignment）和机械可解释性（mechanistic interpretability）。（**机械可解释性可以理解为试图从模型内部的“神经元”和激活模式出发，解释模型为什么会产生某些行为。它不是只看输出，而是研究模型内部发生了什么。**）

比如，如果某些内部激活和“欺骗”相关，我们希望能监测、理解，甚至在训练和产品层面处理它。

第二层是评测（evals）。这更像实验室里的测试。我们把模型放进各种合成场景，看它会不会做危险、错误或不符合预期的事。

第三层是在真实世界里观察。随着模型越来越强，这一层越来越重要。一个模型可能在前两层表现很好，但进入真实用户环境后，问题才会暴露出来。

Claude Code 最早在 Anthropic 内部用了四五个月，才对外发布。因为那时它是一个真正能行动的代理，我们需要确认它足够安全。co-work 也是类似路径：先内部试，再和少数客户试，然后以 research preview（研究预览）的方式发布。

我们也希望整个行业一起把安全标准往上推。所以 Claude Code 团队开源过一个 sandbox（沙箱）工具。它可以把代理限制在一定边界内，避免它访问系统里的所有东西。这个沙箱不只适用于 Claude Code，也适用于其他代理。

我们内部把这种思路叫“race to the top”，也就是让行业在安全和质量上往高处竞争，而不是只比谁更快。

## 怎么更好地使用 Claude Code

如果你刚开始用 Claude Code，我有几个非常具体的建议。

第一，用最强的模型。

现在我会用 Opus 4.6，并且总是开最大努力模式。很多人会本能地选择便宜一点的模型，比如 Sonnet，觉得这样省钱。但更弱的模型可能需要更多 token、更多返工、更多纠正，最后不一定更便宜。

第二，用计划模式（plan mode）。

我大概 80% 的任务都会从计划模式开始。它本质上非常简单，只是在提示词里加一句：“请先不要写代码。”就这么简单。

但效果很好。模型会先和你来回讨论方案。等计划看起来靠谱，再让它执行。到了执行阶段，如果计划足够好，我通常会开启自动接受编辑。尤其是 Opus 4.6，很多任务能一次完成。

在终端里，按两次 Shift+Tab 就能进入计划模式。桌面应用、网页和 Slack 集成里也有对应按钮。

第三，不要只盯着终端。

很多人一提 Claude Code，就想到终端。终端当然很好，我们也支持各种终端。但同一个 Claude Code 代理现在也能在桌面应用、移动端、Slack、网页等地方使用。

不同人有不同工作流。你要做的是找到最适合自己的入口。

## 当速度越来越快，更要用常识

有人问我，我常回到的人生原则是什么。

我的答案是：用常识。

很多工作里的失败，都是因为人没有用常识。他们只是跟着流程走，或者跟着惯性做一个并不好的产品，或者明明感觉哪里不对，却没有停下来重新思考。

我最欣赏的结果，往往来自那些能从第一性原理出发、形成自己判断的人。如果一件事闻起来不对劲，那它大概率真的有问题。

AI 会让执行变得越来越快。正因为执行变快，判断会变得更重要。

Claude Code 这一年的增长让我很兴奋。它已经变成一个很大的业务，也成了很多工程师每天离不开的工具。但从更大的尺度看，我仍然觉得这件事只完成了 1%。

大多数人还没有用过 Claude Code。大多数人甚至还没有真正用过 AI 代理。

我相信 Anthropic 很早就看清了一条路线：先是编码能力，然后是工具使用，再到计算机使用（computer use）。（**计算机使用指的是模型能像人一样操作电脑界面、浏览器和应用，而不仅仅调用后端工具。**）

Claude Code 只是这个路线上的第一个重要产品。

接下来，AI 会继续从“帮你写代码”，走向“帮你完成电脑上的工作”，再走向“帮你判断下一步该做什么”。

这会改变软件工程，也会改变很多别的职业。

我不知道这个未来最终会是什么样。但我很确定：如果每个人都能构建软件，很多今天无法想象的东西会被创造出来。

而我们现在要做的，是尽量让这件事发生得更好、更安全，也更有用。
